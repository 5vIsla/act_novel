# Agent 工具交互借鉴审查

## 审查基线

- 审查时间：2026-05-25
- 审查对象：策划 Agent 页面、底部输入区、运行消息块、上下文抽屉、历史 / 文件 / 模型入口。
- 参考对象：Codex CLI / App 的命令、状态线、权限、fork、compact、status 等交互；opencode 的 modes、commands、permissions；Claude Code 的快捷键、side question、prompt suggestions、transcript viewer、shell prefix、history search。
- 结论：不应照搬终端黑底和高密度 TUI 外观；应继续借鉴“输入区命令化、运行态可控、上下文可追踪、低频功能不挤占主线程”的交互结构。

## 二次审查补充

这次补充审查重点不是继续照抄外观，而是检查 Codex App / Codex CLI / Claude Code / opencode 里还有哪些“交互组织方式”能迁移到小说策划 Agent。结论是：还有一些值得抄，但要落在主输入区、消息流、状态线和 Agent 运行模型里，不能落成新的后台配置面板。

| 优先级 | 新增可借鉴点 | 来源模式 | 适合本项目的原因 | 当前差距 | 建议落地 |
| --- | --- | --- | --- | --- | --- |
| P0 | 持久目标 / Goal 行 | Codex App 有 `/goal`，目标运行时在 composer 上方显示暂停、恢复、编辑、清除。 | 小说策划经常是“完成某本书设定整理 / 写完本章定位 / 修完世界书冲突”这种持续目标，比单条聊天更接近 Agent。 | 已在 composer 上方增加当前目标行，支持编辑、清除和 `/goal 目标文本`。 | 后续可把目标持久化到后端 run/session，而不只是前端本地偏好。 |
| P0 | 命令排队 | Codex CLI 支持运行中输入 slash command 并排队到下一轮。 | 用户在 Agent 运行中常会想到“下一步顺便检查角色卡”或“结束后换模型”，不应只能暂存普通文本。 | 运行中发送已变成暂存下一条，但没有区分命令和普通任务。 | 运行中如果输入 `/` 命令，显示为“下一轮命令队列”，运行结束后按命令语义执行。 |
| P0 | 统一 mention | Codex 新命令里强调文件 mention；Claude / opencode 都支持 `@` 文件。 | 小说项目不仅有文件，还有角色卡、世界书、记忆、正文版本、扮演 transcript，统一 mention 能让用户显式指定证据。 | 文件抽屉存在，但输入区没有一等引用。 | `@` 候选源统一为：本书工作区文件、档案、世界书、记忆、正文版本、扮演记录、上下文资产；发送时只传引用，不塞全文。 |
| P0 | composer 级状态线可配置 | Codex / Claude 都把模型、上下文、目录、权限、任务进度放到底部状态线。 | 这些是长时间写作 Agent 的安全感来源，但不该占据顶部大面积。 | 信息分散在顶部栏、抽屉和工具条。 | 底部状态线显示模型、模式、权限、上下文压力、工作区、分支、后台任务；可在设置里选择显示项。 |
| P1 | 任务列表轻显示 | Claude Code 会在复杂任务时显示 task list，并可切换。 | 策划 Agent 已经有 taskGraph，但默认展开太像运行报告；轻任务条能让用户知道 Agent 在做什么。 | 任务图存在于运行块里，入口偏审计。 | 消息流里默认只露出“当前任务 + 进度点”，展开后看完整 taskGraph。 |
| P1 | Session recap | Claude Code 在用户离开一段时间后显示一行会话回顾，也可手动 `/recap`。 | 用户试用时可能离开，回来后不知道 Agent 做到哪里；比自动关闭或大段历史更友好。 | 现在没有回到页面后的轻量回顾。 | 浏览器标签页重新聚焦且距离上次完成超过阈值时，生成一行本地状态回顾：最后完成、未决审批、可继续动作。 |
| P1 | Transcript viewer / 原始审计查看 | Claude Code 有 transcript viewer，默认工具调用折叠，需要时展开全部。 | 当前项目审计能力强，但抽屉内容偏重；需要一个“查看完整过程”的统一入口。 | 运行块、上下文抽屉、历史抽屉分散。 | 增加“查看本轮完整 transcript”入口，默认只读，不影响主线程；支持定位工具、证据、diff、stdout。 |
| P1 | 项目级自定义命令 / 技能 | opencode 的 `.opencode/commands/*.md`、Claude skills 都能把常用任务做成可调用 prompt。 | 小说项目天然需要“整理设定 / 世界书诊断 / 章节写前定位 / 扮演审查 / 改稿学习”等可复用流程。 | 后端有 skillpack，但用户不能在每本书工作区维护自己的命令。 | 每本书工作区支持 `.novel-agent/commands/*.md`，出现在 `/` 菜单；命令只能生成任务提示或调用已注册工具，不能私自执行 shell。 |
| P1 | Doctor / 配置诊断 | Claude `/doctor`、Codex `/debug-config` 用于解释为什么环境或配置不对。 | 用户经常遇到 provider、模型、baseUrl、工具权限、RAG 索引、工作区路径问题，需要一键诊断而不是猜。 | 有错误分类和部分面板，但缺统一诊断入口。 | `/doctor` 输出本书 Agent 诊断：Provider、模型能力、工具开关、工作区、索引、权限、最近失败原因。 |
| P2 | 最新输出复制 / 摘要复制 | Codex 有 `/copy` 类最近输出复制。 | 用户可能需要把策划结论、角色卡草案、正文片段复制到别处。 | 可以手动复制，但没有结构化复制。 | 在消息旁提供“复制结论 / 复制审计摘要 / 复制正文片段”，不要把它做成主按钮。 |
| P2 | 外部编辑器编辑长输入 | Claude 支持用外部编辑器编辑 prompt。 | 用户会粘贴长设定或长章节要求，网页 textarea 不一定舒服。 | 当前只能在网页输入框里编辑。 | 后续可做“打开本书临时草稿文件编辑后提交”，但 Web 实现成本较高，低优先级。 |

## 不建议照搬

| 功能 | 不建议原因 |
| --- | --- |
| 终端黑底 TUI 外观 | 用户明确不喜欢暗沉风格；小说工作台有背景、材质和长期阅读场景。 |
| 默认强 shell / `!` 无门槛执行 | 小说策划不是代码仓库维护，shell 只能是高级工具。 |
| 大量快捷键优先 | Web 用户可发现性更重要；快捷键应做增强，不应替代可见入口。 |
| 通用无限子 Agent | 小说域更适合固定子 Agent：资料考据、世界书审查、角色一致性、正文风格、扮演整理。 |
| 把所有配置都塞到输入区 | Codex 命令适合终端；本项目仍需要设置页承载 provider、RAG、权限、verifier 等长期配置。 |

## 下一步建议

1. 继续做命令排队、prompt suggestion、历史搜索、side question、审批 once / always / reject、task list 轻显示。
2. 继续做 `/doctor` 配置诊断的后端原生报告、完整 transcript viewer、session recap、项目级自定义命令。
3. 最后做 `!` shell prefix、keymap、导出会话和外部编辑器长输入。

验收标准：这些能力必须围绕主输入区和消息流，不能新增一堆后台卡片；所有写入和工具操作仍走统一工具注册表、权限、diff、checkpoint 和审计。
