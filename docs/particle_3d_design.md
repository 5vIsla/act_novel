# 片状粒子三维翻转设计

## 问题判断

当前 Canvas 2D 方案不能真正解决片状粒子的翻转和倾斜。`scale/skew/transform` 只能做二维仿射变换，视觉结果会变成压扁、斜切或偷偷改变整体方向；它不是花瓣、枫叶、蒲公英种子绕自身表面法线产生的三维姿态变化。

因此片状粒子需要改成真正的三维纹理平面：

- 运动层：粒子位置、速度、出生边缘、标准风、局部意外风、风切变。
- 航向层：整片粒子随气流方向有限摆动，用来表现被风带着走。
- 表面层：同一个纹理平面独立计算 `yaw / pitch / twist`，用来表现翻面、翻起和边缘扭曲。这里不再使用平面内 `roll` 自转，因为它会被误读为贴图自己旋转。
- 图像层：樱花、枫叶、蒲公英使用本地多姿态 sprite 纹理。樱花使用单片花瓣，不使用五瓣花形。多姿态素材只参与内部颜色、明暗和折起提示，最终 alpha 始终来自基础轮廓，避免翻面时突然换成窄长轮廓或透明边缘碎刺。

## 技术方案

### 渲染器

使用 Three.js 的 `BufferGeometry + InstancedBufferGeometry`：

- 每个片状粒子不再是单个 Canvas 2D sprite，而是一个低分段薄片网格。
- 几何由前后两层表面组成，每个顶点带 `sheetFace = 1 / -1`。前层和后层拥有相同 UV，后层索引反向，用 `FrontSide` 让相机只看到真实朝向自己的表面。
- 顶点 shader 根据 `yaw / pitch / twist / zCurve / sheetFace` 把局部点写入真实 `z` 轴，再交给透视相机投影。它不再只是二维 `scale/skew`。
- 相机使用透视相机，并按视口高度反推距离，让 `z = 0` 平面仍然对齐屏幕像素；局部 `z` 深度则会产生真实视差。
- 材质使用透明 `ShaderMaterial`，`premultipliedAlpha`、`depthWrite: false`、`depthTest: false`，避免透明 WebGL 在黑色清屏上产生脏边。
- 四种姿态贴图先合成单张横向图集。shader 一次采样基础、左翻、右翻、折起四个区域，但只混合 RGB 与明暗，alpha 始终使用基础轮廓。这样可以增强翻面感，同时不把花瓣轮廓换成另一张图。

### 坐标层拆分

每帧矩阵按固定顺序组合：

1. `translation`：粒子屏幕位置。
2. `bodyRollZ`：每片粒子出生时获得非累积身体基准朝向，并只允许小幅随风摆动。它不是追随风向或运动方向的 `headingRotation`，不会持续自转，但会打破所有贴图永远竖直向下的问题。
3. `surfaceYawY`：由风压作用点相对质量中心形成的力矩驱动左右翻面。为了不把小花瓣大面积压成线，几何倾斜不能靠固定低频或随机少数触发；当前由横向风载、整体风压、轻部抬升、形状质量差异和物理 `yaw` 连续决定翻面强度。
4. `surfacePitchX`：由风压、质量中心、空气受力中心和惯量共同驱动迎风翻起或压下。
5. `zCurve / sheetThickness`：局部弯面和前后薄片间距进入真实 `z` 轴，用于产生纵向深度、侧缘提示和法线变化。
6. `poseBlend`：根据局部风压把基础、左翻、右翻、折起四种 sprite 的内部颜色混合，但不改变基础 alpha 轮廓。
7. `normalLight`：按表面法线、前后面、弯曲深度和风向计算明暗，让翻面有层次但不变成灰黑碎片。
8. `scale`：粒子实际大小。

不再使用 `headingRotationZ` 让整片贴图朝风向或运动方向平面旋转。贴图仍然必须有自己的身体局部坐标系，否则所有花瓣都会保持屏幕竖直方向，形状场也会被错误绑定到屏幕上下左右。当前实现用 `bodyRollZ` 定义每片粒子的身体朝向，再把真实风压投影进该局部坐标系；风只通过连续质量分布、质心、空气受力中心、协方差惯量、表面 yaw / pitch / twist 和局部顶点形变产生视觉变化。

这能保证“随风摆动”和“翻转倾斜”共存：摆动改变整片粒子的行进姿态；翻转倾斜改变片状表面朝向。

### 风场输入

新三维片状层不再复用 tsParticles 的 shape 绘制器，但会复用同一套配置语义：

- 出现方位决定出生边缘。
- 运动方向决定初始惯性。
- 标准风持续存在，强度在范围内平滑起伏。
- 意外风是局部事件，有位置、半径、方向、强度和持续时间。
- 速度使用目标速度收敛，不允许逐帧无限加速。

### 粒子姿态

对每个粒子维护：

- `yaw`：左右风压造成的表面翻面。
- `pitch`：迎风 / 背风造成的翻起。
- `twist`：非累积颤动信号，参与姿态贴图权重和受限弯面深度；它不能变成平面内自转，也不能直接横向拉伸透明边缘。

所有姿态使用阻尼弹簧或指数平滑，并设最大角速度和最大角度，避免机械急转弯和自转。

翻面计算必须使用粒子自身局部坐标：把风压向量投影到粒子的局部前后轴与左右轴上，再驱动 `yaw / pitch / twist`。不能直接用屏幕 x/y 分量控制翻面，否则会看起来像风在偷偷改变粒子方向。

表面翻转风压只能来自标准风和意外风。出生后的运动方向速度、下落速度或粒子当前速度只影响位置移动，不参与轻边受风判断；否则向下运动会被误当成迎面风，导致花瓣轻端总是竖直向下。

移除下落速度后，视觉响应不能继续沿用旧的二次压缩公式。实现上把真实风压拆成两层：`surfacePressureDrive` 负责轻边前移、翻起和姿态贴图权重，使用接近线性的低风速响应；`windSwayDrive` 负责随风摆动和连续风压波动下的轻微颤动，只由标准风、局部意外风和真实表面风压驱动。这样能保留“向下速度不是风”的约束，同时避免轻部偏向风向和摆动因为风压变小而消失。

重量和风强不能再分散成互不相干的视觉倍率。片状粒子的风响应按近似阻力关系统一为：

`风载比 = 0.5 * 空气密度 * 风速^2 * 阻力系数 * 迎风面积 / 重量`

其中 `重量 = massKg * 9.80665`，用户配置的“物体重量”会同时放大质量、重量和角惯量；同等风速下，重量越大，`windLoadDrive` 越小，位置、翻起、轻边前移和摆动都会自然变弱。贴图轮廓测量得到的 `dragCovariance` 会参与计算 `shapeExposure`：迎风方向看到的横向轮廓越宽，受风面积越大，压力中心偏移、表面翻动和轻部响应也越明显。这比固定判断“上/下/左/右哪边轻”更符合用户要求，因为贴图可以本身是歪的，也可以是蒲公英这种不同材质结构。

### 和现有系统关系

- 雪花、雨继续使用 tsParticles。
- 枫叶、樱花、蒲公英使用新的 `ThreeSheetParticleLayer`。
- 粒子层仍然 `pointer-events: none`，前景粒子不能挡住页面交互。
- 旧的 Canvas 2D 片状 `roleplay-wind-image` 不再保留，避免后续误走二维压扁 / 斜切方案。

## 自检标准

实现完成后必须验证：

- 枫叶、樱花、蒲公英粒子来自 Three/WebGL canvas。
- 同一个粒子同时存在非零 `yaw/pitch/twist`，且没有整片 `headingRotation` 平面转向。
- `yaw/pitch` 不通过压扁 sprite 实现。
- 粒子不会围绕中心绕圈，不会持续自转；`twist` 只能造成局部边缘起伏，不能在 shader 中变成 `rotateZ`。
- 粒子层 `pointer-events: none`。
- 构建通过，真实页面无运行时错误。
- 必须做真实截图验收；如果看起来像碎石、脏点、虫子或不可辨认物，即使调试数据正确也不算完成。
- 樱花花瓣截图不能偏灰暗、棕紫或高饱和玫粉；在当前蓝粉背景上也应保持低饱和、偏白的淡樱花粉。
- 樱花花瓣不能只做平移；调试样本里的 `yaw / pitch / poseWeights` 必须达到肉眼可见的幅度，并通过连续截图确认位置和姿态都有变化。
- 不能把姿态链路做成“数值非零但视觉不可见”。`yaw/pitch`、姿态贴图权重、`bodyRoll` 摆动、`lightDeform` 和 `sheetShape` 局部位移都必须映射到足够明显但不脏、不碎、不自转的画面变化。
- `sheetShape` 不能直接把透明边缘顶点大幅横向拉动。局部形状场位移必须被 alpha 可见轮廓限制，主要用于微小局部前移、法线和明暗变化；侧翻、折起和受风姿态主要交给姿态贴图与三维表面倾斜，避免出现毛刺和撕裂。
- 随风摆动的主来源必须是标准风和局部意外风的风压分量。相位噪声只能作为很弱的颤动附属项，不能主导花瓣姿态，否则会看起来像花瓣自己摆动。
- 樱花花瓣运动需要使用轻量物理模型：每片花瓣有 `mass`、`centerOfMass`、`angularInertia` 和 `windResponse`，气流作用点相对重心形成力矩，再由角惯量、阻尼和回正力积分成 `yaw / pitch / twist`。不能回退成“目标角度弹簧 + 随机旋转”的假摆动。
- 单点重心不足以解释花瓣哪里更容易被风掀起。每片花瓣还需要从贴图轮廓采样得到连续质量矩、拖曳矩、局部覆盖率、局部拖曳密度、边缘 / 薄区程度、局部风载响应、`stiffness` 和 `edgeFlex`。不再保留 `rootTipBias`、`leftRightBias` 这类离散尖根 / 左右字段，因为它们会把贴图重新硬拆成固定方向。
- `weightDistribution` 不应是随机姿态参数。对同一材质，质量分布应来自片状轮廓的面积矩：哪边轮廓面积或体积更大，哪边更重；材质层提供 `densityKgM3`、`thicknessMm`、`projectedAreaMm2`、`massKg` 和 `weightNewton`。轻的一侧在局部风压下拥有更高响应，重的一侧提供更强惯量和回正。
- 形状差异不能只在力矩公式里留下系数，还必须有可见局部形变：局部风压命中覆盖率低、拖曳 / 质量比高、边缘或薄区明显的位置时，该区域顶点应沿风向前移、沿视线抬起并改变局部法线。这个动态量由 `sheetShape` 和 `instanceLightDeform` 共同驱动，而不是靠全片 `yaw/pitch` 伪装。
- 透明贴图不能依赖黑色透明像素参与线性采样。三维姿态图集生成后需要做透明像素颜色扩散，保留 alpha 不变，只把邻近有效花瓣颜色扩散到低 alpha 像素，避免 WebGL 缩放采样产生脏边或黑边。
- 透明 WebGL 画布不能用普通 alpha 在黑色透明清屏上累计颜色，否则边缘半透明像素会被黑色污染。片状粒子 shader 输出必须使用预乘 alpha，renderer 和 material 也要匹配预乘 alpha 混合。

## 非互斥重构约束

上一版把“随风翻动”“轻部局部受风”“折叠姿态贴图”和“轮廓保持”都放进同一条顶点变形链，结果必然互相冲突：只要风压变大，`zCurve`、局部 slip 或折叠贴图 alpha 就可能把花瓣拉长、压扁或替换成窄条。新的实现必须先把层级拆开：

1. **轮廓层不可拉伸**：屏幕轮廓只来自 `local.xy * instanceScale` 和整体三维姿态投影。风场、形状场、轻边受风不得再直接给屏幕 XY 坐标加大幅偏移。整体三维姿态可以造成透视压缩，但不能让任一方向超过原本局部尺寸。
2. **弯曲层不改轮廓**：`zCurve` 只参与深度明暗、法线和翻面感，不再先写入顶点 z 再被 `rotateX/rotateY` 投影回 XY。否则弯曲越明显，屏幕上越会被拉长。
3. **形状受风不改 alpha 轮廓**：`sheetShape` 可以影响局部法线、明暗、edge lift、轻部响应强弱，但不再横向拖拽透明边缘。要表达“轻部更受风”，优先使用局部高光、阴影、边缘抬升和连续姿态状态，而不是把顶点拉出去。
4. **姿态贴图不改外轮廓**：多姿态贴图只能混合颜色和内部明暗，最终 alpha 以基础花瓣贴图为主。`left/right/fold` 的 alpha 不能替换外形，否则翻面会看成突然变成另一张窄长花瓣。
5. **方向状态连续**：风压方向必须有自己的连续单位向量状态。低风压时沿用上一帧方向；局部意外风出现或消失时，只改变气流向量，再由阻尼系统逐步传给 `yaw/pitch/twist/lightDeform`，不能每帧把瞬时风向直接归一化后塞给姿态。
6. **所有视觉量同源**：位置、表面风压、姿态角、轻部响应和姿态权重都从同一个连续风压状态派生。不能让位置跟一套风走、翻面跟另一套风走、贴图权重又跟相位正弦走。

这意味着片状粒子的功能不再互相排斥：

- “不拉伸”由轮廓层保证。
- “翻转倾斜”由整体 surface yaw / pitch 的投影压缩、法线明暗和连续姿态权重保证。
- “轻部更受风”由质量矩和形状场改变局部法线、深度明暗、edge lift 和 lightDeform 强度保证。
- “随风摆动”由连续风压状态驱动，而不是靠自转或随机相位主导。

如果后续要做更强的真实局部弯曲，应该升级到有边长约束的低分辨率薄片网格，让每条边通过约束保持长度；不能再回到直接给平面顶点叠加风向位移的做法。

## 尺寸、出生范围和姿态的硬边界

本轮把三个容易互相干扰的链路重新定边界：

- `size` 是最终屏幕外接尺寸上限，单位是 px。片状粒子会先用贴图长宽比反推出实际 plane 高度，保证任意身体朝向和三维倾斜后的外接尺寸都不超过用户设置的值。
- `spawnEdges` 是独立保存的出生边缘配置，不再退回 `spawnPosition` 后丢掉范围。出生范围只在粒子创建或重生时决定初始坐标，不能参与风向、运动方向或姿态计算。
- `yaw / pitch / twist / lightDeform` 主要改变整体三维投影、法线明暗、姿态贴图颜色和边缘抬升提示。`zCurve` 可以用很小的受限深度进入三维投影，制造真实弯面视差，但不能再沿风向直接拖拽透明边缘。这样“能看出翻转”和“不拉伸”不会互相冲突。

## 本轮形状场受风实现

旧实现的问题不是没有风载数据，而是风载没有被转成“按贴图形状分布的局部位移”。旧的 `localLightAdvance/localEndAdvance/instanceAerodynamics` 会把风响应再次摊到较宽区域，后续的 `lightPole` 思路又过于接近“找一个轻端 / 尖端”。这些路径已经清理，shader 现在从 `sheetShape` 读取每个顶点的局部形状场，再结合 `instanceLightDeform` 的真实风载方向和强度计算局部形变。

当前实现核心为三层：

- `shapeField` / `sheetShape`：从真实贴图像素采样得到每个顶点的局部覆盖率、局部拖曳密度、边缘 / 薄区程度和局部风载响应。它不是固定尖端，也不是上下左右分区，而是跟随整张贴图轮廓的连续形状场。
- `lightDeform`：每帧由真实气流、风载比、受风面、物体重量共同驱动，作为 `instanceLightDeform` 传入 shader。它只表示当前风的方向、强度、抬升和扰动，不决定哪块区域一定受影响。
- `windShapeManifold`：在 shader 内按 `sheetShape`、真实风向、质量张量距离、边缘柔度和下风投影形成局部流形。不同风向下，受影响区域会根据形状场重新分布；覆盖率低、拖曳 / 质量比高、边缘或薄区明显的位置响应更强，主体区域响应更弱，避免整片花瓣被一起平移。

本轮同时修正物理基准：同一粒子预设内的物理长度、宽度、厚度、投影面积、体积、质量、重量、阻力面积、角惯量和基础风响应只来自原始材质配置与原始贴图轮廓。每片粒子的屏幕高度、宽度、透明度和出生相位仍可随机，让画面自然，但这些视觉缩放不能反向参与真实面积、体积、质量或风载计算。换句话说，“看起来大 / 小”不是“真实大 / 小”。

这个实现仍然遵守原约束：

- 不恢复 `headingRotation`。
- 不把“禁止 headingRotation”理解成“没有身体朝向”；每片花瓣有非累积 `bodyRoll`，但它不对齐风向、不持续自转。
- 不把下落速度、出生运动方向或粒子当前位置速度当作表面风。
- 不用固定上 / 下 / 左 / 右或“真实尖端”判断受风区；即使贴图本身是歪的，也由 `sheetShape` 和当前风向共同决定局部响应。
- 物体重量会同时压低 `windLoadDrive`、`lightDeform.drive` 和 `lightDeform.lift`，而不是只影响调试字段。
- 显示缩放不会改变同类粒子的 `physicalMassKg`、`projectedAreaMm2`、`dragAreaM2` 或基础 `windResponse`；只有用户配置的重量倍率会统一改变质量、重量和角惯量。

## 参考依据

- NASA Glenn 的阻力方程资料使用 `D = Cd * 0.5 * rho * V^2 * A` 描述阻力与空气密度、速度平方、参考面积和阻力系数的关系；本项目的 `windLoadDrive` 采用同一结构，再除以粒子重量得到风载比。
- NASA Glenn 的形状阻力资料说明不同形状会改变阻力系数和有效受风面积；本项目因此用贴图轮廓的拖曳协方差计算 `shapeExposure`，而不是只用一个固定面积。
- NSF 对 Ristroph 纸片 / 纸飞机研究的介绍强调，重量位置和质心会影响纸片下落控制与 flutter/glide 表现；本项目继续保留质心、压力中心和角惯量链路。
- Li、Goodwill、Wang、Ristroph 关于纸片 / 纸飞机的研究指出，重心位置会改变薄片从 flutter、tumble 到 glide 的运动模式，准定常气动模型需要考虑动态压力中心和气动力矩。
- Belmonte、Eisenberg、Moses 的 falling paper 研究使用惯性阻力和升力解释纸片从 flutter 到 tumble 的转变，说明单纯路径插值不足以表现自然薄片下落。
