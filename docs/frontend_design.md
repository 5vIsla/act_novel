# 前端详细设计

## 1. 设计修正

旧版三栏工作台能验证“扮演到改写”的技术链路，但不适合真实长篇创作。问题不只是页面拥挤，而是入口心智错了：用户打开一本小说时，首先需要和 AI 一起梳理灵感、设定、人物和大纲，而不是立刻面对一堆模型和角色表单。

新版前端采用多页面结构：

1. 小说库：新建、打开、删除小说。
2. AI 策划：打开小说后的默认页面，用对话沉淀角色卡、背景、大纲、场景、线索。
3. 档案：查看和人工修正 AI 提取出的结构化项目档案。
4. 扮演配置：由 AI 根据档案生成配置草案，用户可改 JSON 或表单后采纳。
5. 行文：在配置稳定后运行多 AI 扮演，并将扮演记录改写成正文。
6. 长期记忆：审查、编辑和整理稳定长期事实，只处理会持续影响后续创作的记忆条目。
7. 世界书：独立维护 World Info / Lorebook 条目，控制关键词触发、插入位置、范围和可见性。
8. Agent 设置：集中配置 Agent 权限、工具开关、RAG 检索、向量索引和 verifier runner。
9. 提供商：配置 baseUrl、接口类型、key、模型查询和手动模型。

## 2. 页面信息架构

```text
┌───────────────┬─────────────────────────────────────────────┐
│ 小说侧栏       │ 当前页面                                      │
│ 新建/打开/删除 │ 小说库 / AI 策划 / 档案 / 记忆 / 世界书 / 扮演配置 / 行文 / 提供商 │
└───────────────┴─────────────────────────────────────────────┘
```

侧栏只负责项目级入口，不再承载角色卡和提供商编辑。具体工作放进对应页面，避免把创作、档案、配置和行文混在同一屏。

## 3. 页面设计

### 3.1 小说库

职责：

1. 新建小说。
2. 打开小说。
3. 删除小说。
4. 展示每本小说的角色数、策划消息数和正文段数。

打开小说后直接进入“AI 策划”页。

### 3.2 AI 策划页

职责：

1. 作为小说打开后的默认界面。
2. 做成 AI 聊天对话窗口：顶部显示策划 AI 身份，中间是连续消息流，底部是输入组合框。
3. 作为强制挂载“小说策划 skill”的持续对话工作台，而不是配置表单或独立提取工具。
4. 档案摘要和扮演草案只作为右侧上下文侧栏，不挤占聊天主流程。
5. 用户可以输入灵感、人物关系、世界规则、章节目标，也可以直接粘贴旧大纲、设定文档、章节片段等长文本。
6. 策划助手 AI 像正常对话一样回应用户，同时输出结构化 `archivePatch`。
7. 前端展示对话历史和本次同步进档案的内容。
8. 提取档案、沉淀记忆、更新世界书和生成扮演配置都是策划 Agent 的工具能力，由用户在对话里提出目标后触发，不再放成页面顶部流程按钮。
9. 输入区就近提供策划模型快速切换；上下文、历史、本地文件检索等辅助能力放入互斥抽屉，不破坏主线程。
10. “运行审计与上下文”抽屉只承载本轮运行观察：工具审计、证据调度、上下文包预览、压缩摘要和上下文资产。
11. Agent 权限、工具开关、检索配置、向量索引和 verifier runner 是跨 Agent 配置，统一放入“Agent 设置”页，不放进策划线程审计抽屉。

这个页面参考 `webnovel-writer` 和 `my-novel-skill` 的思路：用户通过持续交互和不断补充材料，让 AI 逐步拟定角色卡、背景、大纲、场景和线索，再进入具体行文。它不是普通聊天页，因为 Agent 拥有检索、编辑、写入和审计工具；它也不是一次性提取工具，因为长篇策划需要反复讨论、修正和追问。不是每次对话都要写入档案、记忆或世界书，写入必须由 Agent 判断确有必要后通过 `archivePatch` 或 `skillOps` 主动执行。

### 3.3 档案页

职责：

1. 编辑小说标题、核心命题、背景、当前时间、当前地点、剧情方向、叙事基调。
2. 查看和编辑角色档案、场景档案、线索档案。
3. 支持人工查看和修正策划 Agent 已沉淀的档案。

第一版用 JSON 文本区编辑角色、场景、线索数组。这不是最终体验，但比堆很多固定字段更灵活，适合先验证“AI 提取档案”的结构是否够用。

### 3.4 扮演配置页

职责：

1. 展示 AI 生成的扮演配置草案。
2. 用户可以直接采纳最新草案。
3. 用户可以修改 JSON 后采纳。
4. 用户可以“采纳并启动扮演”。
5. 配置策划助手、导演、次要角色群、改写器的提供商、模型、温度。
6. 管理实际参与扮演的角色卡。

设计原则：

1. 策划阶段不要求用户先配齐所有角色。
2. 行文前才要求扮演配置完整。
3. 次要角色仍不逐个配置模型，保持“次要角色群 AI”统一扮演。

### 3.5 行文页

职责：

1. 运行下一轮扮演。
2. 将最近扮演改写成小说正文。
3. 清空当前行文会话。
4. 展示导演引导、主要角色输出、次要角色群输出。
5. 展示正文片段和最近事件。
6. 正文片段显示草稿、已采纳、已废弃状态。
7. 草稿可以编辑、采纳或废弃；采纳后进入稳定正文库和 RAG 证据层。
8. 已采纳正文再编辑时，会让旧 AI 整理记忆过期，再从新正文中提取长期事实。
9. 支持导演 AI 生成章节级写前定位，写前定位只提供软统筹，不替角色决定本轮选择。
10. 支持审查最新扮演和最新正文，审查结果显示严重程度、问题、证据和建议。
11. 支持单角色重跑，重跑时保留导演引导和其他角色输出。

行文页只处理具体写作阶段，不再混入前期策划。

### 3.6 记忆页

职责：

1. 展示长期记忆总数、有效、暂定、过期、冲突等统计。
2. 新增、编辑、删除结构化记忆项。
3. 清理旧版机械投影记忆。
4. 调用 AI 整理近期扮演和已采纳正文中的长期事实。
5. 明确展示写入边界：长期记忆只保存稳定、可审计、带可见性的事实。

记忆页不是“备注页”。它的内容会进入实际 AI 调用，所以必须显示状态、来源、可见性和证据，便于用户审查污染。

不放在记忆页的内容：

1. RAG 检索配置、embedding/rerank 模型、向量索引重建。
2. RAG 检索证据和本轮上下文包预览。
3. 工具审计、证据调度、压缩摘要和上下文资产。

其中检索配置、embedding/rerank 模型和向量索引重建属于跨 Agent 配置，放在“Agent 设置”页；RAG 证据和本轮上下文包预览属于运行观察，放在 AI 策划页的“运行审计与上下文”抽屉。

### 3.7 世界书页

职责：

1. 维护独立 World Info / Lorebook 条目，不再只依赖角色 `lorebook` 字段和项目上下文包间接覆盖。
2. 支持名称、启用状态、范围、所属角色、关键词、二级关键词、正则、始终注入、优先级、单条预算和递归扫描。
3. 支持可见性：策划、导演、改写器、次要角色群、全部主要角色或指定主要角色。
4. 支持插入位置：角色卡前、角色卡后、记忆前、记忆后。
5. 统计全部条目、启用条目、角色/私有条目和单次触发上限。
6. 支持导入和导出常见酒馆 World Info / Lorebook JSON，导出时保留本项目的范围、可见性和触发方式扩展信息。
7. 角色卡导入时会把 `character_book` 同步为角色私有世界书条目，用户可以在本页继续编辑或禁用。
8. 世界书页负责维护条目本体；触发结果、扫描文本和最终组包顺序在 AI 策划页的“运行审计与上下文 / 上下文包”里按预览对象查看。

角色扮演阶段会优先使用接近酒馆的 `tavern_context`：固定角色卡、触发世界书、角色可见记忆、最近可见历史。策划和导演仍使用全局统筹型 RAG，不应该简单照搬酒馆式关键词触发。

### 3.8 Agent 设置页

职责：

1. 配置 Agent 权限模式：只读、自动编辑低风险、高风险询问、全自动。
2. 配置工具开关：联网、Shell、自定义工具、MCP。
3. 配置命令前缀授权和后台 Shell 作业管理。
4. 配置 Verifier Runner，包括确定性判定、外部命令检查和模型审查器。
5. 配置 RAG 检索模式、embedding/rerank 槽位、证据预算和向量索引重建。

这个页面是项目级运行边界，不是某条策划消息的审计记录。独立成页可以避免用户误以为权限、检索和 verifier 只影响当前策划对话。

### 3.9 提供商页

职责：

1. 新增或编辑提供商。
2. 配置 `baseUrl`、`responses` 或 `chat/completions`、`key`、模型查询路径。
3. 查询模型。
4. 手动添加和删除模型。
5. 远端模型查询只显示候选列表，不会直接保存全部模型；用户需要逐个点击“添加”。

key 只由本地后端保存，前端只显示脱敏结果。

## 4. 前端状态

前端只维护 UI 状态：

1. 当前页面。
2. 当前选中的角色 ID。
3. 当前编辑中的提供商 ID。
4. 当前加载状态。
5. toast 消息。
6. 当前编辑中的记忆项和正文片段。
7. 模型查询候选列表和候选筛选词。
8. 当前编辑中的世界书条目。
9. 策划 Agent 运行审计抽屉里的预览对象、查询词和临时上下文包预览。
10. 策划输入草稿，用来保证页面重绘时不丢失长文本。
11. 当前策划 Agent runId，用于显示运行状态和发送终止信号。

业务状态以后端 `/api/state` 为准。每次写入成功后以后端返回的 state 重新渲染，避免前后端分叉。

模型查询结果、候选筛选词、当前编辑中的记忆项、当前编辑中的正文片段属于临时 UI 状态，不写入后端。真正保存的只有用户明确点击保存、添加、采纳或废弃后的结果。

## 5. 核心流程

### 5.1 策划到档案

1. 用户打开小说。
2. 默认进入 AI 策划页。
3. 用户输入灵感、问题或粘贴长文档。
4. 策划 skill 先理解输入，再以自然对话方式回复。
5. 策划 skill 同时返回 `archivePatch` 和 `archiveDiagnostics`。
6. 后端只把满足写入合同的稳定内容合并到 `archives`，空值和无主键数组记录不会覆盖旧档案。
7. 聊天消息可以展开查看本轮同步档案和写入诊断，用户可据此人工修正或重新整理。

### 5.2 档案到扮演配置

1. 用户在策划线程里要求 Agent 准备、更新或检查扮演配置。
2. 策划 Agent 根据档案、角色卡、世界书、长期记忆、RAG 检索证据和可用提供商生成配置草案。
3. 草案写入 `planning.roleplayDrafts`，并在扮演配置页可见。
4. 用户在扮演配置页审阅。
5. 用户可直接采纳，也可修改 JSON 后采纳。

### 5.3 扮演配置到正文

1. 采纳配置后，后端同步 `scenario`、`aiRoles` 和实际角色卡。
2. 用户进入行文页。
3. 运行下一轮扮演。
4. 导演 AI 生成场景约束。
5. 主要角色 AI 分别扮演。
6. 次要角色群 AI 统一扮演。
7. 改写 AI 将扮演记录改写成正文。
8. 正文先进入草稿状态，用户审阅后再采纳或废弃。
9. 采纳正文后，正文才进入稳定记忆和后续检索。

## 6. 可运行验证

本地模拟提供商覆盖以下链路：

1. AI 策划对话。
2. 档案提取。
3. 扮演配置生成。
4. 采纳配置并启动扮演。
5. 扮演记录改写为正文。
6. 正文采纳、废弃和记忆整理。

因此没有真实 key 时，也能验证产品主流程。真实模型接入后，输出质量由模型和提示词共同决定。

## 7. 设计系统落地

当前项目是原生 HTML/CSS/JS 工作台，不引入 Ant Design、TDesign、Arco 或 Semi 组件库。前端采用自定义 Design Token，借鉴企业级中后台的信息密度和状态完整性：

1. 一个主视觉体系：灰蓝底色、白色工作面板、绿色主操作、棕色创作提示、红色风险操作。
2. 统一组件约定：侧栏、项目头、工作流导航、状态条、面板、列表项、统计块、聊天气泡、表单、JSON 编辑器和 toast 都走同一套圆角、边框、阴影和密度。
3. 打开小说后顶部展示状态条，直接显示策划消息、档案、记忆、世界书、扮演轮次和正文采纳情况。
4. 导航不再只是标签文字，每个工作流入口都有二级说明，帮助区分“策划对话、稳定档案、RAG 记忆、世界书触发、演员配置、扮演行文”。
5. AI 策划页保持聊天为主工作区，档案摘要和扮演草案作为右侧上下文，不抢主流程。
6. 移动端和窄屏会把侧栏、项目头、状态条和策划布局降为单列或双列，没有横向滚动。
7. 运行态验证覆盖桌面 1440px 和移动 390px 视口，关键元素可渲染，且没有横向溢出。

## 8. 第二版工作台重设计

本轮按 `frontend-design-systems` 流程先做设计证据矩阵，再改代码。前端定位不再是“多页面表单”，而是“剧本调度台”：每一本小说打开后，都应能看到当前 Agent 运行、资料质量、检索证据、扮演配置和行文流水线。

### 8.1 设计证据矩阵

| 项目 | 设计决策 |
| --- | --- |
| 产品类型 | 长时间使用的 AI 小说创作工作台，包含对话式 Agent、资料库、Lorebook、RAG、扮演调度和正文产出。 |
| 目标用户与频率 | 作者和策划者高频使用，需要反复导入资料、修档案、跑 Agent、审查正文、回退重发。 |
| 主设计体系 | 自定义 Design Token + 高密度工作台，不引入外部组件库。 |
| 参考来源 | Ant Design 的导航与企业后台模式；Semi Design 的 token、AI 组件和反馈状态；Arco Design 的企业级工作台与主题完整性；SillyTavern 的角色卡、世界书触发和上下文预算业务模型。 |
| 可借鉴模式 | 左侧项目导航，中间主任务区，右侧上下文、证据和审计；列表页使用操作栏、结构化条目、详情编辑器；Agent 页展示运行、工具调用、回退、终止和重发。 |
| 不采用内容 | 不做营销 Hero，不做大面积紫蓝渐变，不做白卡片堆叠，不把所有功能挤到一页，不用“聊天窗口 + 一排按钮”冒充 Agent 工作台。 |
| 页面模式 | 小说打开后默认进入策划 Agent 对话；资料、世界书、记忆、扮演和行文分别成为可审计工作页；每页都有页内指挥栏。 |
| 组件策略 | 工作流轨道、页内指挥栏、状态条、Agent 运行板、Inspector、资料记录列表、编排图、行文流水线、风险检查列表。 |
| 视觉记忆点 | “剧本调度台”：用流程编号、状态边线、触发词、角色槽位、检索证据和审查链形成业务识别。 |
| 状态覆盖 | 正常、空、运行中、风险、保存中、禁用、可回退、向量过期、草案待采纳、正文草稿。 |
| 响应式策略 | 桌面三栏或双栏，窄屏转单栏；工作流轨道在移动端横向滚动，页内指挥栏改为纵向操作。 |

### 8.2 已落地结构

1. 小说头部改为工作流轨道，策划、档案、记忆、世界书、扮演、行文都有编号、说明和状态。
2. 每个小说页新增页内指挥栏，集中显示当前页目标、关键状态和主操作。
3. 策划页新增 Agent 运行板和运行审计抽屉，暴露运行、证据检索、档案写入、人工确认和版本回退状态。
4. 档案页改成“结构化源 + 档案质量 Inspector + 记录列表”，不再只是大文本框和卡片。
5. 扮演页新增 AI 编排图，明确策划、导演、主要角色、次要角色群和改写器的关系。
6. 行文页新增流水线，显示导演定位、多角色扮演、一致性审查和正文草稿的推进状态。
7. 样式层加入第二版工作台 token 和响应式规则，避免普通后台模板感。

### 8.3 设计自检

这版仍保留原生 HTML/CSS/JS 和现有后端接口，因此不会因为视觉重设计破坏保存、查询、导入、运行等主流程。它解决的是信息架构问题：用户打开小说后先面对策划 Agent 和工作流状态，而不是被所有配置字段淹没。

仍需后续继续增强的地方：

1. 档案数组仍是 JSON 源编辑，后续可以做成可局部编辑的结构化 Drawer。
2. 世界书和记忆页已有列表和编辑器，但还可以增加更明确的命中预览、向量陈旧原因和批量操作。
3. Agent Inspector 现在展示后端已有报告，后续可进一步做成真正的工具调用时间线。

### 8.4 第三版视觉修正

用户指出第二版仍然平平无奇，这个判断成立。原因不是缺少“设计系统名字”，而是落地时只做了信息架构和白卡片式状态组件，没有形成该产品自己的视觉概念。大厂设计系统不能直接等于高级感；如果没有真正引入组件库主题，也没有把业务隐喻转成视觉语言，最后仍会像普通后台。

第三版把视觉方向收束为“导演剪辑台 / 剧本调度台”：

1. 左侧改为深色创作工程坞，和主工作区形成明确空间分工。
2. 主区改为暖色剧本纸画布，使用细网格和纸面色，而不是灰白后台背景。
3. 项目头部加入彩色压线和更强标题层级，像打开一个剧本工程。
4. 页内指挥栏改为黑色控制台，集中呈现当前 Agent 目标、模型、回退和主操作。
5. 状态条改为仪表条，风险、空状态和运行状态都有明显色块，不再像普通 KPI 卡片。
6. 策划对话区域保留聊天功能，但背景、消息、输入区都转向剧本纸与控制台风格。
7. 审计区域强调证据检索、工具执行、人工确认和回退，不再使用“预检”这种难理解的前端按钮文案。
8. 扮演编排图和行文流水线使用不同角色色，强化“多 AI 演员调度”的业务辨识度。
9. 移动端保持小说内主工作区在前，侧栏后置；指挥栏在 390px 下改为单列，避免挤压。

第三版仍不引入外部 UI 库。原因是当前项目是原生 HTML/CSS/JS，如果强行迁移 Ant、Arco 或 Semi，成本会超过本轮重设计范围，也容易把业务逻辑拖进组件迁移风险。这里采用的是设计系统思想：token、密度、状态、布局、反馈、响应式和可审计工作台模式，而不是复制组件库外观。

### 8.5 第四版结构重设计

第三版仍然保留了旧区域逻辑：左侧项目列表长期占据小说内首屏，顶部状态和指挥栏继续挤压策划对话，导致“策划 AI 对话”虽然被强调，但并没有成为真正主画布。这不符合用户最初的核心要求：打开一本小说后，首先应该是 AI 交互页面。

第四版不再继续叠视觉覆盖，而是替换小说内整体 shell：

1. 小说内页面改用 `studio-shell-v4`，不再复用旧的 `workspace-v2 + sidebar + header + status` 结构。
2. 左侧项目列表退出小说内首屏，改为 72px 窄工作坞，只保留工作流导航。
3. 策划页改为 `planning-canvas-v4`，桌面 1280px 下主画布约占 73% 屏宽。
4. 右侧 Inspector 收窄为 220px，只保留档案、证据、回退摘要，不再和主对话抢空间。
5. 策划页标题、模型快速切换、运行状态、skill 状态、消息流和输入框都放在同一张主画布内，形成连续的 AI Agent 对话工作区；生成扮演配置由 Agent 工具调用完成。
6. 其它小说页面通过 `studio-page-v4` 适配新 shell，保留页内动作和状态，但不再使用旧大页头。
7. 移动端主画布优先显示，导航坞下沉到页面底部，避免小说内首屏先出现项目列表。

第四版的设计重点从“看起来像一个主题”转向“空间权重正确”：策划 AI 对话是第一工作区，资料和证据只是辅助面板。只有当用户进入档案、世界书、记忆、扮演、行文页面时，对应工具才获得主空间。

## 9. 自检

### 是否解决用户指出的问题

解决。新版不再把所有功能挤在一屏；小说打开后进入挂载策划 skill 的 AI 对话页，配置和行文分开。策划阶段不再被设计成“点按钮提取档案”的工具，而是持续对话过程中自动沉淀档案。

### 是否偏离扮演法核心

没有。扮演法被放在具体行文阶段，符合“先策划档案，再行文扮演”的工作流。

### 仍然不足

1. 档案页的数组编辑暂时用 JSON，后续应做成更友好的卡片表单。
2. 扮演记录还不能逐条人工编辑；单角色重跑已经落地。
3. AI 提取档案还没有“用户确认后再入库”的审核层，后续应加入采纳/驳回机制。
4. 正文已有草稿、采纳、废弃状态，但还没有多版本分支对比。

## 10. 第五版 Agent 工作台修正

这次修正针对“策划 AI 仍像普通 AI 聊天窗口”的问题：

1. 策划页保留线程式 Agent 主体，不再把“提取档案”和“生成扮演配置”放成顶部流程按钮。
2. 生成扮演配置、检索档案、编辑记忆、编辑世界书等都进入 `skillOps` 工具合同，由策划 Agent 在同一轮任务里规划、调用、观察和收束。
3. 输入区新增策划模型快速切换，下拉直接作用于当前小说的 `aiRoles.planner`。
4. “预检”前端文案改为“查看检索上下文”；执行前证据检查仍保留在后端审计里，但不再作为用户必须理解的抽象按钮。
5. “恢复上一条”改为用户消息旁的“回到此处编辑”；版本抽屉只负责运行历史、checkpoint 和写入回退。
6. “Agent 工作区与本地文件”作为互斥抽屉接入本地 Node 后端，主职责是查看或调整当前小说的 Agent 工作区；新建小说时后端会自动在项目内分配 `novels/<书名>-<小说id>/` 作为默认工作区，这个工作区等同 Codex 打开文件夹后的 workspace / cwd，额外资料文件夹只是补充来源，关键词检索、读取文本和系统打开文件只作为人工预览和调试入口。
7. 策划页重新纳入项目材质体系，跟随背景、透明度、模糊和材质模式，避免变成独立黑色终端窗口。

## 11. Agent 工作区与工具

本地文件能力不能设计成“用户先手动搜，再把结果交给 AI”。正确心智是：

1. 每本小说创建时会自动保存自己的 `planning.defaultAgentFolder`，这是当前 Agent 的工作区，等同 Codex 打开文件夹后的 workspace / cwd；抽屉只负责查看或更换它。
2. `planning.localFileSources` 只表示额外资料文件夹，不能被理解成 Agent 的唯一电脑权限。
3. 策划 Agent 在同一轮任务里根据用户目标自行调用 `searchLocalFiles` 和 `readLocalFile`，默认从当前小说 Agent 工作区开始。
4. Agent 工作区之外的读取、扫描、写入和 shell cwd 必须请求用户明确允许或进入人工确认；移动、删除等更危险操作暂不开放默认工具。
5. 工具调用结果进入消息下方的工具审计和运行轨迹。
6. 抽屉里的“人工预览”只用于用户检查目录是否配对、文件是否能读，不是主工作流。

这能保持 Codex/opencode 式交互：用户给目标，Agent 自己查资料、读取证据、再决定是否写入档案、世界书、记忆或扮演配置。

## 12. Agent 过程展示修正

策划 Agent 的运行过程不应占满对话框，也不应被表现成固定“三步循环”。前端呈现规则调整为：

1. 消息主区域只显示用户任务、Agent 回复和一行工具摘要。
2. 运行过程默认折叠，只显示事件数、工具数、模型调用数和最新状态；详细时间线需要用户展开。
3. 工具计数按真实工具调用计算，包含项目检索、本地检索、读取文件、写入记忆、编辑世界书、编辑档案、生成扮演配置等，不再只统计写入成功数。
4. 后端只有内部运行保护预算，不把“最大步数”作为前端流程标签展示；预算耗尽时显示为可继续的暂停，而不是完成或失败。
5. 停止按钮必须同时处理活跃 Agent 和服务重启后遗留的陈旧 running 记录；陈旧运行应清理为已终止，不能让页面一直显示运行中。
6. 策划线程默认只加载最近一段历史消息；上滑到顶部附近时静默请求更早消息并保持滚动位置，不使用显眼分页按钮，也不一次性把所有历史对话渲染出来。
7. 压缩状态不能占用独立对话框。已完成 run 的事件、token 和压缩信息进入对应 Agent 回复下方的折叠审计；只有运行中、等待确认、失败且没有回复、或保护暂停且需要继续入口时，才显示运行状态块。
8. 页面滚到很下方时，最上方页面栏和策划页“上下文 / 历史 / 文件 / 模型”入口不能彻底失联。原本页面栏保持正常结构，不改成线程内折叠栏；桌面端额外提供视窗顶部热区，鼠标悬浮到浏览器窗口最上沿时，临时唤出页面栏和策划工具栏的副本。这个热区必须通过 Portal 挂到 `document.body`，避免被页面材质层的 `backdrop-filter`、`transform` 或局部滚动容器改变 fixed 定位参照。移动端不依赖 hover，继续使用原页面内控制区。

## 13. 上下文压缩展示原则

前端不把上下文压缩当成用户必须处理的消息流。压缩属于 Agent 内部上下文管理：

1. 正常完成的压缩只显示为轻量状态或折叠审计，不生成新的“压缩对话框”。
2. 只有压缩失败、上下文溢出、需要用户确认丢弃某类证据，或达到运行保护预算时才打断用户。
3. 历史对话展示分页和模型上下文压缩是两件事：前者解决页面性能，后者解决模型输入预算。
4. 更接近 Codex/opencode 的方向是“保留最近尾巴 + 摘要旧上下文 + 裁剪大工具输出 + 保留可追溯引用”，而不是把完整历史塞回模型。
5. 运行审计抽屉可以查看 `planning.contextCompaction` 摘要、token 统计、版本链、质量审查和上下文资产数量，但必须明确它只是模型上下文缓存，不代表写入档案、记忆或世界书。
6. “上下文资产”页签只做轻量审计：展示 asset id、类型、标题、token 和来源，不把大工具原文直接铺到界面；后端资产索引只保存短预览，完整 payload 作为旁路文件保存，Agent 需要原文时通过 `searchContextAssets` 定位并用 `readContextAsset` 工具读取。
7. “压缩摘要”页签展示质量审查、source range、token 统计和版本链；旧版本回滚只回滚内部压缩缓存，不回滚档案、记忆、世界书或消息正文。

## 14. 流式输出与 Verifier 展示

策划 Agent 运行块里的实时面板承担“观察运行中状态”的职责，不替代最终回复：

1. 模型 token 流默认折叠显示，只用来观察当前模型是否还在输出。
2. 持续 shell、工具 stdout/stderr 和 verifier 命令输出归入同一个实时输出面板，避免占用主对话气泡。
3. 运行头部显示 `已用 token / prompt 预算`，悬浮时展示上下文窗口、输出保留和预算来源，让用户知道这是模型画像预算，不是简单字符估算。
4. Verifier Runner 折叠项展示确定性检查、命令检查和模型审查器结果；多模型审查器用步骤列表呈现，不把它做成自动化仪表盘。
5. 这些信息默认属于运行审计层，不应该自动写入记忆、世界书或档案。
