# 酒馆、角色卡与本项目设计记录

## 参考资料

- SillyTavern 角色设计文档：<https://docs.sillytavern.app/usage/core-concepts/characterdesign/>

## 可转化为工程规则的结论

1. “酒馆”类应用的核心不是直接写小说，而是通过角色卡、世界信息、历史消息和系统提示组织一段角色扮演上下文。
2. 角色卡的价值在于把角色稳定性拆成可复用字段：名称、描述、性格、场景、首条消息、示例对话、系统提示、后置历史指令、标签和角色专属世界书。
3. Character Card V2/V3 把角色本体和扩展信息分离，适合本项目保存为结构化角色资料；当前项目已支持 JSON 导入/导出，也能读取 PNG 文本元数据中的 `ccv3`、`chara`、`ccv2` 载荷，导入为主要角色时会自动绑定项目可用 AI 槽位。
4. 对小说写作而言，角色扮演输出不应直接当成最终正文。更稳的流程是：导演 AI 先给出场景约束，角色 AI 在约束内扮演，最后由改写 AI 将多方输出整理成小说段落。
5. 主要角色单独调用模型可以强化角色个性；次要角色共用一个 AI 可以降低编排复杂度，但必须把“次要角色群”也当成一个独立演员，而不是硬编码对白生成器。
6. 提供商接口需要抽象成两种常见形态：`chat/completions` 和 `responses`。模型查询则尽量走 OpenAI 兼容的 `/models` 或用户配置的查询路径。
7. 角色定义会占用上下文预算，所以角色卡不是越长越好。长期稳定信息放档案和角色卡，本轮真正生效的信息应在行文前被压成更短的执行约束。
8. 酒馆支持多角色或群聊，但本项目不能直接照搬“群聊触发顺序”。小说写作需要导演先约束场景，再让角色在同一约束下行动，最后由改写器统一成正文。
9. World Info / Lorebook 的工程价值在于“条件触发注入”：条目可以按关键词、过滤条件、插入顺序、插入位置和上下文预算进入 prompt。
10. Data Bank / Vector Storage 的工程价值在于“文档级 RAG”：资料先被切块、向量化，再只召回相关片段，避免把长文档整段塞进上下文。
11. Chat Vectorization 的工程价值在于“历史消息 RAG”：从很久以前的聊天里召回相关消息，放到模型更容易注意的位置。
12. 酒馆文档也明确提醒向量召回质量依赖 embedding，结果不完全可预测；本项目如果需要确定性，不能只依赖向量，应保留关键词、结构化档案和人工审阅。
13. 角色扮演 AI 更适合接近酒馆的上下文触发：角色卡固定层、世界书关键词扫描、最近可见历史和少量私有记忆；策划 AI 和导演 AI 不适合照搬这种方式，因为它们需要全局统筹、事实生命周期和冲突裁决。

## 记忆与 RAG 结论

本项目的长期能力应采用“结构化记忆为事实真源，RAG 为证据召回层”的设计。

1. 策划 AI 需要全局创作决策记忆：用户意图、否定过的方向、已确认档案、待定问题、导入文档证据。
2. 导演 AI 需要统筹记忆：时间线、场景状态、世界规则、伏笔、角色当前可见状态、本轮禁止事项。
3. 角色 AI 需要私有主观记忆：角色卡、自己经历过的事件、自己知道的事实、关系判断、最近互动。
4. 改写 AI 需要忠实改写记忆：本轮扮演、最近正文尾部、文风约束和不可新增设定。
5. 角色可见性必须在检索前过滤，不能先全局召回再靠提示词要求模型“不要知道”。
6. 当前项目采用统一记忆底座和分策略组包：策划是 `project_rag`，导演是 `orchestration_rag`，主要角色是 `tavern_context`，次要角色群是 `group_tavern_context`，改写器是 `faithful_adaptation`。
7. 详细设计见 [memory_rag_design.md](./memory_rag_design.md)。

## 本地项目对照

用户提到的 `E:\novel\webnovel-writer` 和 `E:\novel\my-novel-skill` 更接近“先档案、再执行”的创作系统，而不是一次性生成器。本项目采用其中适合扮演法的部分：

1. 先通过 AI 策划对话沉淀项目档案，避免用户一打开小说就被模型参数和角色表单打断。
2. 档案不是正文素材池，而是边界和稳定事实；行文前只把本章或本轮真正要生效的内容压进扮演配置。
3. 角色卡用于稳定角色的反应模型和说话方式，场景、线索、背景仍保留在项目档案里，避免每个角色卡塞进整份世界观。
4. 具体行文时再启动扮演法，让导演、主要角色、次要角色群和改写器各自承担边界清楚的职责。

## 本项目采用的编排结构

1. 导演 AI：负责背景、时间、地点、场景压力、剧情目标和禁止事项。
2. 主要角色 AI：每个主要角色有独立提供商、模型和温度，并使用角色卡字段构造系统提示。
3. 次要角色群 AI：统一接收全部次要角色卡，只输出本轮需要出现的次要角色行动。
4. 小说改写 AI：只使用导演约束和扮演记录写小说正文，不重新决定世界规则。
5. 前端控制层：管理小说、角色、提供商、模型、扮演轮次和改写结果。

## 自检

- 该结构能达成“角色分别扮演，再改写成小说”的核心目标。
- 主要角色和次要角色的计算边界清晰，不靠增加大量状态字段模拟角色差异。
- 提供商抽象面向接口形态，而不是针对某个厂商写死。
- 当前版本已经实现 OpenAI 兼容接口、Character Card JSON 导入/导出、PNG 元数据导入、独立世界书编辑器、常见世界书 JSON 导入/导出、按任务区分的记忆组包策略、本地向量索引、embedding/rerank、导演写前定位、审查链和单角色重跑。下一步更有价值的是把审查结果转成可视化差异、把向量库替换为更适合超长项目的专用存储，以及让章节级目标和正文版本管理形成闭环。
