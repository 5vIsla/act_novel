# Codex 式 Agent 体验根因验收台账

本文档只记录“为什么反复修同类问题”和“以后如何避免继续循环”。它不是功能愿望清单，也不是宣传完成度；它是后续修改策划 Agent 体验时必须先对照的验收台账。

## 为什么之前会循环

之前的问题被拆成了太多表象，例如“过程不像 Codex”“命名奇怪”“编辑有 bug”“完成后不收纳”“权限批准后没继续”“会话切换慢”。这些表象背后常常是同一批根因：

1. 运行事件、工具事件、审计事件、上下文资产事件没有统一成一条用户可理解的动作生命周期。
2. 主消息流、独立 run 块、运行历史抽屉和 transcript 抽屉之间存在多套展示 fallback，修了其中一套，另一套仍会露出旧样子。
3. 前端状态同步里同时存在乐观消息、后端分页、SSE / 轮询补拉、编辑重发首包，任何一处覆盖策略不对，就会出现“发出后消失、刷新才出现、短暂显示两个”等症状。
4. 后端 Agent 循环虽然有工具和多步能力，但展示层仍会把底层 item / event 当报表展示，导致看起来不像 Codex 的“行动段落”。
5. 权限、失败、阻断没有全部作为 observation 回到 Agent 修正循环；有些路径直接变成 UI 终止态，用户就会觉得批准后没有后续。
6. 设计修改没有固定验收标准，每次根据最新截图局部修补，导致“越改越不像”的感觉。

以后不能再按这些表象单独修。每次修改必须先判断它属于下面哪个根因，再改对应入口和测试。

## 六个根因验收项

| 编号 | 根因 | 用户可见验收 | 代码验收 |
| --- | --- | --- | --- |
| C-01 | Agent 运行循环不像 Codex | 同一轮里能根据模型输出调用工具，工具结果回流，再继续模型调用或收束；失败先给 Agent 修正机会，不能直接静默中断。 | 后端 run 必须记录模型调用、工具 observation、修正轮次和 stopReason；工具失败不能只生成 terminal item。 |
| C-02 | 工具过程展示未聚合 | 主消息流只显示“读取文件 / 写入档案 / 检查通过 / 等待权限”这种一条动作；开始、完成、失败和结果属于同一条。 | 前端只从统一的 public process step / display part 渲染，不直接展示原始 `item`、`event`、`assetRef`、`runId`。 |
| C-03 | 终态收纳和运行中展开不一致 | 运行中能看到当前动作；完成后自动收成一行摘要和“过程详情”；失败 / 等待权限才展开关键原因。 | `PlanningMessageRunTrail`、`PlanningRunMessageBlock`、transcript fallback 的 live / settled 策略必须一致。 |
| C-04 | 会话状态覆盖和编辑重发不稳定 | 用户消息发送后立即显示且输入框清空；编辑某条消息是在该消息上编辑，从该处继续；旧后续不能污染当前路径。 | 乐观消息、分页回包、编辑重发首包、branch/fork 合并必须走同一 merge 函数，并有测试覆盖。 |
| C-05 | 权限批准后不能自然继续 | 权限条主动、轻量、可见；批准后原 run 继续，拒绝后 Agent 能换路径或说明阻断；同作用域只读授权不重复问。 | approval result 必须写回等待中的 tool call，并触发原 run resume；权限缓存按工具、路径、会话 / 小说工作区作用域匹配。 |
| C-06 | 上下文资产和历史证据暴露底层名 | 用户要求读当前文件时显示“读取文件内容”，不是“查找历史证据 / 读取上下文资产”；旧 run 引用只在详情中说明。 | evidence scheduler、readContextAsset、readLocalFile 的 display name 由 source intent 归一化；底层引用只进 debug 详情。 |

## 不再重复的工作规则

1. 每次用户指出“又不像 Codex”时，先把现象归到 C-01 至 C-06；如果归不进去，新增根因项，而不是新增一条同义 bug。
2. 同一个根因没有通过页面或 API 验证前，不允许把它从“风险”改成“完成”。
3. 修展示时必须同时检查主消息流、独立 run 块、运行历史、过程详情和 transcript，不允许只修一个入口。
4. 修编辑 / 会话时必须检查发送、刷新、切换会话、编辑重发、派生、回退和删除会话，不允许只检查普通发送。
5. 修权限时必须检查“请求出现、批准、原 run 继续、拒绝、重复授权不再询问”完整链路。
6. 修工具命名时必须检查用户明确读文件、自动读历史证据、读上下文资产、复用旧文件资产四种路径，避免又把当前文件显示成旧证据。

## 当前完成度基线

| 根因 | 当前判断 | 说明 |
| --- | --- | --- |
| C-01 | 风险 | 后端已有多步 Agent、工具和修正基础，但还需要用真实运行证明失败 observation 能回到 Agent，而不是部分路径直接终止。 |
| C-02 | 风险 | 已有聚合组件，但仍要检查所有 fallback 是否绕过聚合层。 |
| C-03 | 风险 | 最近已修终态重复展示，但需要真实页面确认运行中、完成、失败三态都一致。 |
| C-04 | 风险 | 最近修了编辑重发首包覆盖，但还要补前端状态合并测试和真实点击验证。 |
| C-05 | 风险 | 已有权限机制，但用户反馈批准后仍失败，说明 resume 链路或 UI 状态还有未覆盖路径。 |
| C-06 | 风险 | 第一层文案有清理，但 evidence / context asset 的显示意图还要端到端验证。 |

## 下一步验收顺序

1. 先用真实页面复现一次普通发送、编辑重发、运行完成收纳，确认 C-03 / C-04 当前状态。
2. 再用一个需要读文件的会话验证 C-02 / C-06：主消息只出现“读取文件内容”，详情才出现引用和旧证据。
3. 再用一个需要权限的外部路径验证 C-05：批准后原 run 必须继续，不继续就修后端 resume。
4. 最后验证 C-01：构造工具失败或 schema 修复场景，确认 Agent 得到 observation 后会修正，而不是直接停止。

只有 C-01 至 C-06 都通过页面验证和自动测试，才允许说“这轮 Codex 式体验验收通过”。不能说“已经完全复刻 Codex”。
