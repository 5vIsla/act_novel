# 用户思路与灵感

- 做一个利用扮演法写小说的项目。
- 主要角色都单独利用 AI 扮演。
- 次要角色共同用一个 AI 进行扮演。
- 背景、时间、地点、剧情引导单独一个 AI。
- 最后再利用 AI 将扮演内容改编写成小说的一部分。
- 需要深入研究“酒馆”和角色卡。
- 前端可以新建小说、删除小说、控制扮演。
- 前端可以添加主要角色和次要角色，并且单独选择提供商和模型。
- 前端可以配置提供商、baseurl、responses 或 chat/completion、key。
- 前端可以查询和添加模型。
- 前端不应该把全部内容挤在一个页面里，应该多设计几个页面。
- 打开一本小说后的界面应该先是 AI 交互页面。
- 这个 AI 交互页面应类似 `E:\novel\webnovel-writer` 和 `E:\novel\my-novel-skill` 里的 skill 或插件流程。
- 用户通过不断交互，让 AI 提取和拟定角色卡、背景、大纲等内容。
- 只有到具体行文阶段，才运用 AI 扮演法。
- AI 可以生成扮演配置，用户可以直接由 AI 启动扮演，也可以人工修改后再启动。
- AI 策划本质上应该是一个与 AI 持续交互、共同设计小说的对话界面。
- AI 策划对话界面需要强加一个类似 skill 的策划流程。
- 用户会不断和 AI 对话，也会提供一大段文档。
- AI 需要在持续对话过程中不断根据输入提取角色卡、背景、大纲等需要的内容。
- AI 策划界面应该做成一个 AI 聊天对话窗口界面。
- skill 或写成插件该如何写，需要参考提供的一个 skill、一个插件，以及“酒馆”。
- AI 策划和导演 AI 都需要强大的记忆和统筹能力。
- 角色 AI 也需要强大的记忆能力。
- 项目需要考虑 RAG。
- 需要参考 `E:\novel\webnovel-writer`、`E:\novel\my-novel-skill` 和“酒馆”的记忆与 RAG 做法。
- 需要支持 SillyTavern 角色卡 PNG 元数据导入，不能只支持 V2 JSON。
- 需要独立的 World Info / Lorebook 编辑器，不能只靠角色 lorebook 和项目记忆包间接覆盖。
- 扮演 AI 与策划 AI、导演 AI 可能需要不同检索方式；角色扮演希望更接近“酒馆”的上下文触发方式。
- 需要接入 embedding、rerank 和 vector DB。
- 章节级写前定位是导演 AI 的工作，用来对扮演进行基本统筹安排，但不能对角色扮演做过多约束。
- 需要审查链来保证一致性等问题。
- 需要支持单角色重跑。
- 策划 AI 的 skill 需要跟上：能从对话和长文档中提取需要内容，严格规范如何写入资料、设定、角色、场景、线索等档案，并具备纠错、冲突提示和资料沉淀所需功能。
- 生成或融合世界书时，条目数应尽量对齐来源，不能用规则矫正硬凑。
- 世界书“对齐来源”不是只保留单一来源；只要是真正属于绝区零的资料，都应该尽量保留，不能因为对齐原始文件而丢掉其它已确认来源的条目。
- 英文来源的绝区零世界书也要保留，但必须做中文化处理；专有名词、姓名、组织名、店铺名等要尽量符合《绝区零》简中官方或通行中文写法，不能长期保留半英文、暂译、待校状态。
- 生成世界书可换用其他模型，但不要使用 Pro 模型；可以使用 CaMeL 提供商的 deepseek-v4-flash。
- 策划 skill 需要增强写入、编辑和检索能力，不能只做一次性档案同步；它应能检索已有档案、记忆、世界书和角色卡，并通过可审计操作更新资料。
- 策划 skill 需要能检索和编辑独立世界书、记忆条目本体、已生成正文版本、扮演记录、删除档案条目和深层嵌套内容；其中高风险对象需要安全边界和审计。
- 策划 AI 更像 Codex、Claude Code、opencode 等 AI 智能操作工具：需要先检索上下文，再执行受控操作，但权限必须受项目后端白名单和 preflight 约束。
- 策划 AI 希望更接近真实 Agent：能在同一轮里多步循环，先查一下，根据结果继续查、编辑，再检查是否需要修正，最后收束回复；不能用 4/6 这种低步数上限把它做成固定按钮流程。
- 策划 Agent 需要真实产品级控制：运行中可以终止；完成后可以回到上一步；可以把上一条用户发送内容放回输入框，编辑后重新发送；也可以回退后直接重发。
- 策划 Agent 需要进一步接近真实 Agent 工具：支持 AI 调用自动重试、失败原因诊断、工具失败后的自动修正、结构化错误分类、模型输出修复重试、异步任务队列、同小说写入锁、流式或实时步骤显示、每步 checkpoint、单步回退、高风险操作人工确认、自检闭环、任务计划与完成判定、Token / 上下文预算管理和 Agent 运行历史。
- 前端不能把策划 AI 当成普通 AI 聊天窗口；策划页要像 Codex、opencode、Claude Code 一样是交互式 Agent 工作台。
- 策划页不应使用“发送策划给 AI”这种普通聊天逻辑；用户是在给 Agent 下达任务，Agent 会规划、检索、执行、检查、必要时请求确认。
- 回退重写不应割裂成独立按钮，而应并入运行历史、checkpoint、恢复上一条指令和重新运行的 Agent 版本链路。
- 策划 Agent 前端不应该做成自动化仪表盘；更应该像 Codex、opencode 那样，以对话线程为主体，Agent 的计划、检索、工具调用、审批、checkpoint 和结果都嵌在消息流里。
- 策划 Agent 的主输入应像 Codex 底部输入框：用户持续提出后续变更或新任务，Agent 在同一线程中处理，而不是进入单独的自动化控制台。
- 提取档案和生成扮演配置不应依赖外部按钮；它们应该是策划 Agent 自身可根据用户指令调用的工具能力。
- 策划 Agent 页面视觉不能变成不搭项目整体材质的纯黑窗口；应继续服从项目背景、材质、透明度和模糊配置。
- 策划 Agent 线程里需要快速切换策划模型。
- “预检”这种抽象按钮不清晰，应改成能理解的检索上下文查看能力。
- “恢复上一条”不符合 Codex/opencode 交互，应改成消息旁的“回到此处/重新编辑”一类线程内操作。
- 本地文件检索和查看很重要；优先利用本地后端读文件，不必为了这个功能立刻把项目迁移成桌面壳。
- 背景系统需要区分“全网页滚动背景”和“仅视窗背景”，不能默认所有幻灯片都固定在全视窗。
- 前景模块材质不应所有材质共用同一套透明、模糊、染色参数；每种材质需要有自己的配置项，且视觉差异要明显。
- 幻灯片设置里不应长期展示大量“已保存 / 已载入”条目，素材多时最多显示当前预览和数量统计。
- 幻灯片本地绑定需要支持多个文件夹地址，并可设置随机播放或顺序播放。
- 前景透明度下限不能太高，需要允许更透明的前景模块。
- 媒体背景不应显示“已保存图片 / 已保存媒体”这类列表文本，而应直接给出可点击放大的预览。
- 背景预览图必须完整显示，不能裁切，并且可以点击打开大预览。
- 媒体和幻灯片必须共用同一套“仅视窗 / 全网页”逻辑：仅视窗背景固定在视窗内不随滚动变化；全网页背景属于页面本身并随页面滚动，不能用重复图片硬填充。
- 背景系统里的“纹理”功能应替换成“粒子”功能。
- 粒子功能需要支持雪花、雨、枫叶、樱花等效果，并能配置密度、风场等参数。
- 粒子效果不应和媒体背景、幻灯片背景互斥；应能作为独立叠加层叠在当前背景上。
- 选择粒子效果不应该放弃或切换当前媒体背景、GIF、视频或幻灯片背景。
- 粒子效果需要区分后景粒子和前景粒子，两层可以分别设置不同效果和参数，并且可以同时存在。
- 粒子效果不应该塞在媒体或背景设置内部，需要移动出去，成为单独的粒子效果设置入口。
- 粒子设置不应该移到背景页面之外；它应该仍在背景页面中，但作为独立的粒子设置分区或页签，不能塞进媒体设置里。
- 粒子的“前景”指粒子覆盖在正常侧边栏、文本框、页面内容等前台 UI 之上。
- 粒子效果需要像前景材质一样支持边调边看效果。
- 粒子需要可设置出现方向，例如从上往下、从左往右等。
- 风场需要可设置两个坐标来控制方向。
- 风场混乱程度需要能选择发散或聚集等模式。
- 当前粒子效果太粗糙，需要参考成熟粒子方案的做法，不应长期停留在粗糙 CSS 粒子。

- 本地文件检索不应该只是用户手动检索的抽屉功能；它应该成为策划 Agent 自身可调用的工具能力。用户只配置授权资料根目录，Agent 在任务中自行检索和读取需要的本地资料。

- 策划 Agent 不能一直卡在运行中，停止按钮必须真正终止或把陈旧运行状态清掉。
- 策划 Agent 不应该被前端表现成固定“三步循环”；运行过程应像 Codex 一样按真实工具调用自然推进，安全上限只能作为后端保护，不应成为用户心智。
- 策划 Agent 后端可以保留内部运行保护预算来防止失控消耗，但预算耗尽必须表现为可继续的暂停，不能被标记成已完成。
- Agent 过程应默认自动折叠和省略，不能占用过大的对话框；需要时用户再展开查看细节。
- 工具计数必须反映真实工具能力，包括本地文件检索、读取文件、项目检索、档案写入、世界书、记忆、扮演配置等，不能显示为零。
- 策划 Agent 线程不能一次性显示所有历史对话；默认只显示最近一段，用户向上滚动时再像 Codex 一样静默载入更早对话。
- 策划 Agent 的上下文压缩不应该占用一个独立对话框；压缩应是内部上下文管理，把旧对话、工具结果和资料证据变成摘要、档案、记忆或检索索引，只在需要时给用户查看状态。
- 策划 Agent 压缩逻辑应该参考 Codex/opencode：保留最近对话尾巴，旧内容形成可更新摘要，过大的工具输出要裁剪或替换成引用，压缩后继续任务而不是让用户手动处理。
- 不是每一次策划对话都要写入档案、记忆或世界书；这些写入都应由策划 AI 判断确有必要后主动调用工具或显式写入通道完成。
- 策划 Agent 页面滚到很下方时，原本页面最上方栏和“模型 / 上下文 / 文件 / 历史”等工具栏可能离开视野；此时应能把鼠标悬浮到浏览器视窗最上方位置，临时唤出这两条栏，而不是把策划线程内部顶部栏改成悬浮折叠展开。
- 本地文件夹选择不是给 Agent 的唯一授权范围，而是当前小说 / 当前 Agent 的 Agent 工作区；每一本小说都应有自己的 Agent 工作区。它就相当于 Codex 里打开的文件夹，是当前 Agent 的默认环境 / workspace / cwd。Agent 可以具备操作电脑文件的能力，但工作区之外的读取、写入、移动、删除等操作必须经过明确允许或人工确认。

- 粒子的“前景”必须是真正最前面的视觉层，应该覆盖普通侧边栏、文本框和页面内容，而不是只比部分页面模块层级高。
- 粒子设置项需要可折叠，尤其是出现方向、流场和扰动等配置，避免全部平铺挤在一起。
- 风场不应再使用起点和终点两个坐标来表达；更合理的是由出现方向决定主方向，再用流场强度、侧向风、发散/聚集强度和随机扰动等连续值控制。
- 发散和聚集不应是互斥模式按钮，而应该是同一个正负强度值：正值发散，负值聚集。
- 雨的粒子必须保持近直线、有统一方向的雨线，不能被随机扰动弄成歪七扭八的乱线。
- 粒子视觉需要参考成熟粒子方案的发射器、力场、数量限制和 Canvas 绘制思路，不能长期停留在粗糙的 CSS 粒子或简陋形状。
- 粒子效果必须是持续不断的，不应该一阵一阵出现或像批次播放。
- 粒子系统应优先使用成熟粒子项目或引擎，不要继续依赖手写粒子物理一点一点补。
- 粒子形状应尽量支持外部或本地图源，例如雪花、雨滴、枫叶、樱花等 sprite，而不是全部由代码临时画出来。
- 前景粒子必须视觉上在最前面，但交互上必须完全穿透，不能阻挡按钮、输入框、抽屉、弹窗或其它页面操作。
- 粒子风场不应该被表现成固定在中间的单个圈，也不应该变成几个旋转的局部力场圈；用户要的是模拟风本身，即连续气流、主风、侧向风、阵风、切变和湍流，预览应表现为流线和风带，而不是节点圆圈或发散 / 聚集圈。
- 粒子运动绝不能围着画面中心绕圈；实现上不能启用 tsParticles 的 `move.spin` 中心环绕，也不能用基于中心点的横向推拉力来伪装风。
- 粒子本体不应该自转来假装有风；风应表现为粒子被连续气流带走，参数含义要能通过鼠标悬浮说明看懂。
- 策划 Agent 需要继续向 Codex、opencode 一类真实 Agent 靠近：补通用工具层、原生工具调用协议式的后端工具注册表、细粒度权限系统和更强运行时可观测 / 自动修复能力。
- 策划 Agent 需要具备受控的通用 shell / bash 工具，能自己跑命令、检查环境、调用脚本，但 Agent 工作区之外的危险命令必须经过权限控制。
- 策划 Agent 需要具备通用文件系统工具，包括 read、grep、glob、list、edit、write、patch 等，并且这些工具必须服从当前小说 Agent 工作区和权限模式。
- 策划 Agent 需要支持子 Agent、通用 websearch / webfetch，让它能主动联网找资料，但默认仍应优先开放小说域工具，避免安全边界失控。
- 策划 Agent 需要考虑 LSP / 结构化本地资料索引能力，以及真正流式输出，让每一步工具调用即时显示。
- 工具参数失败后不应只是进入 skipped，而应强制进入诊断和修正回合，让 Agent 根据错误类型主动改参数、重试或明确阻塞。
- skillOps 需要升级成后端注册式工具表，统一 schema、权限、执行、展示、重试策略。
- 需要增加全局 Agent 权限模式：只读、自动编辑低风险、高风险询问、全自动。
- 本地文件检索需要接 embedding、rerank 和 vector DB，让 searchLocalFiles 支持语义检索。
- 需要增加 SSE 等真正流式事件通道，而不是只靠轮询刷新运行状态。
- 需要预留 MCP / 自定义工具接口，但默认只开放小说域工具。
- 粒子不能因为禁止自转就变成完全僵硬的贴纸；花瓣、叶片、雪花等应能随当前风向调整朝向，并由气流产生轻微摆动，但这种摆动不能是粒子本体持续自转。
- 粒子风场不能让粒子越跑越快；风应该表现为稳定气流和有界摆动，速度必须收敛在合理范围，而不是每帧累加加速度。
- 粒子的出现方向和风向必须分开：出现方向只决定从哪个边缘进入画面，风向才决定粒子后续被气流带向哪里。
- 风力不能因为做了限速就变得几乎没有影响；风应该明显改变粒子运动方向，但速度仍然需要有界收敛。
- 粒子各项数值调整必须有明显视觉反馈，尤其是粒子本身下落 / 移动速度，不能滑块变化很大但画面几乎没变化。
- 随机扰动不能只有单一强度值，需要能设置扰动强度范围和扰动变化时间范围。
- 每本书对应的档案、世界书、记忆、扮演配置、行文审查等项目资料，应该能放到这本书自己的默认 Agent 工作区里，方便 Agent 像打开项目文件夹一样检索和操作。
- 这里说的每本书默认文件夹，不是指用户手动选择一个任意资料文件夹，而是每创建一本小说，项目内部都应该自动创建一个固定对应的小说文件夹。
- 记忆不能变成随便写入的摘要堆；它应该服务于后续策划、导演、角色扮演和改写，必须有写入约束、证据、状态和可见性。
- 项目需要明确说明借鉴“酒馆”处理记忆的方式，尤其是角色卡、世界书关键词触发、上下文预算、向量召回和角色可见性隔离。
- 粒子系统里的“扰动”应理解为不遵循标准风向的意外风，而不是标准风上的小噪声；意外风需要可设置强度范围和持续 / 切换时间范围。
- 标准风力应做成范围式，并增加标准风频率范围来控制标准风强弱变化频率；阵风强度控件应删除。
- 标准风频率范围必须能设置小数；意外风也需要独立频率范围，且每次意外风的位置和风向都应随机变化。
- 粒子设置里的“出现方向”应拆成“出现方位”和“运动方向”：出现方位决定从哪个边缘进入，运动方向决定出生后的基础运动惯性，二者都应和标准风向分开。
- 前景材质的透明度、磨砂、染色、饱和等参数变化必须明显，拉到最低时不能还残留固定高光、固定边框和固定阴影。
- 粒子“运动方向速度”只应该作用于粒子本身的运动方向惯性，不应该直接放大全局风速；标准风、侧向风、意外风和本体速度混合时要更平滑，避免方向和速度变化生硬。
- 扮演配置等每本书自己的配置，需要让最近成功保存的配置成为这本书的默认配置；单纯生成草案不能刷新默认配置。
- 鼠标悬浮唤出的页面栏、工具提示等内容不能被前景粒子层或其它视觉层盖住。
- 粒子设置页面的悬浮说明不能同时显示两套说明；需要保留一套清晰提示，删除重复提示，并且提示不能太大、太黑或显得突兀。
- 策划 Agent 需要持续 shell 会话、stdout 实时流、命令分段确认、环境探测、后台进程管理、退出码策略和命令级 allowlist 持久授权。
- 策划 Agent 的文件编辑需要更接近 Codex 的成熟编辑流：unified diff、apply_patch、多文件补丁、冲突定位、patch preview 和 patch revert。
- 策划 Agent 的联网检索需要更可靠：可配置搜索提供商、结果缓存、来源可信度、失败重试和引用管理。
- 策划 Agent 需要真正的执行函数体系，不能只停留在预留接口。
- 策划 Agent 需要真正 token 级流式输出，并优先支持模型 token streaming 和工具 stdout streaming。
- 策划 Agent 需要真正任务图、依赖关系、完成判定器、外部 verifier 和自动 resume 队列。
- 策划 Agent 权限需要更细：按目录、按命令前缀、按工具、按会话、按一次性或永久允许的权限策略。
- 原生工具调用协议加严格 schema 是优先级最高的 Agent 稳定性升级。
- 更强 patch / diff / 预览 / 回退系统对世界书、角色卡、正文草稿和项目文件编辑都很重要。
- MCP / 自定义工具白名单要真正落地，但默认仍只开放小说域工具，外部工具必须逐项注册。
- LSP / 结构化资料索引值得升级，尤其是每本小说工作区会放大量 Markdown、JSON、角色卡、世界书和旧稿时。
- 子 Agent 系统适合做资料考据 Agent、世界书审查 Agent、角色一致性审查 Agent 和正文风格审查 Agent，不应该一开始做成无边界通用子 Agent。
- 粒子系统里的“侧向风”命名不准确，它本质上应该是标准风方向的微调，并且应该做成可调范围。
- 粒子频率不应主要用用户难以理解的小数 Hz 展示，界面上更适合用“周期 / 间隔秒数”表达。
- 意外风需要明确是位置随机、方向随机的临时风，而不是固定中心力场。
- 粒子的“出现方位”必须真正控制出生位置，不能让粒子在画面中间凭空出现。
- 粒子标准风和意外风的周期 / 间隔可调范围不能卡在 0.7 秒/次一类偏窄范围；用户希望能调到 0.01 秒/次，同时界面应提示这等价于 100Hz，可能更接近高频抖动而不是自然风。
- 风增强、减弱和消失时，对粒子速度方向的影响应该体现为速度分量连续变化，而不是目标方向突然切换导致粒子生硬转弯。
- 粒子的风场时间必须尽量按用户设置的真实秒数执行，不能因为逐帧计算或频率换算导致体感时间明显不准。
- 意外风应该是局部随机事件：有随机位置、影响半径、随机方向、随机强度、随机持续时间，并且同一画布可以同时存在多个意外风，而不是一个影响全局的随机方向。
- 风场范围参数的语义要统一：强度、周期、持续时间、方向微调等范围要么在范围内随机抽取一个值，要么只在范围内连续变化，不能跳出范围或被固定成单值。
- 标准风语义上应该是持续存在的主气流，强度只是在范围内起伏；不能在低强度段让粒子运动方向重新完全主导，造成风像停了一样、粒子快速变回垂直下落。
- 意外风需要能真实命中随机位置附近的粒子；如果事件存在但影响太弱或半径太小导致看不出来，就没有达到“局部随机风”的目的。
- 策划 Agent 的上下文压缩不能停留在字符数估算；需要更接近模型 tokenizer 的 token 预算。
- 策划 Agent 的大工具结果需要引用化落盘，Agent 后续能按引用读取，而不是把大段结果一直塞进消息或 prompt。
- 策划 Agent 的压缩摘要需要版本链、source range、diff 或回滚依据，不能只有单份覆盖式摘要。
- 策划 Agent 需要压缩质量审查器，能判断摘要是否遗漏用户偏好、稳定事实、未决问题、工具证据和风险。
- 记忆层、档案层、世界书层、运行审计层之间需要更成熟的分层召回，让继续任务时能主动按需读历史证据。
- 策划 Agent 的完成判定不能只停留在后端确定性 verifier；需要独立外部测试进程或第二模型审查 Agent，但恢复队列不应后台无限自动重跑，以免静默消耗模型费用或反复写资料。
- 策划 Agent 需要 Codex 式 verifier runner，把后端确定性判定、外部脚本/项目检查命令、审查模型和最终完成判定统一编排成可配置 verifier 链。
- 樱花花瓣这类片状粒子需要能随风产生翻面变化：不是持续自转，而是跟随气流方向、轻微摆动，并通过视觉上的压缩、倾斜和明暗变化表现被风翻动。
- 粒子效果需要增加蒲公英，作为可选择的独立粒子预设。
- 片状粒子的翻转不是自转，而应根据风朝某个方向和某个倾斜角度轻微或大幅变化；可以考虑给粒子重心。
- 片状粒子的倾斜和翻起不能被做成平面转弯；风造成的翻起应表现为沿受风方向的压缩、错位、明暗和轻微位移，而不是额外旋转。
- 用户不需要手动点“查历史”。
- Agent 应该自动知道该查哪段历史。
- 大量历史不应该堆在聊天窗口里。
- 继续任务时 Agent 应该主动读旧证据。
- Agent 读了什么、为什么读，至少要能在审计层追踪。
- 不应该把所有内部上下文管理都展示成一堆对话框。
- 长期记忆、RAG 检索证据和上下文包不能混放在一个页面里；长期记忆页只管稳定事实，RAG 检索配置、证据调度和上下文包预览应该放在 Agent 运行时上下文位置，并明确给哪个 Agent 使用。
- 策划 Agent 页的审计模块不应该包含不属于策划 Agent 当前运行审计的配置模块；Agent 权限、工具开关、RAG 检索配置、向量索引和 verifier 这类跨 Agent 配置应放到独立 Agent 设置页。
- Agent runtime 需要接入模型原生 tools/tool_choice 协议，skillOps 只作为 fallback。
- Agent 需要自动历史证据调度器，彻底移除“手动查历史”的主流程。
- Agent 工具需要统一注册表，schema、权限、执行、重试、展示和审计都从一个入口管理。
- Agent 运行模型需要细化到 message、turn、item 级，让工具调用、文本、审批、checkpoint 和证据读取成为同一类可追踪事件。
- 小说域 RAG 需要强化分层召回：档案、世界书、记忆、正文、审计和上下文资产要按层路由。
- 子 Agent 需要是真实 session 或 background task，不能只是一次普通 AI 调用。
- Agent 需要 E2E 测试覆盖启动任务、自动检索、写入记忆、回退、取消、失败重试和压缩续跑。
- 策划 Agent 主输入区要像 Codex / opencode 这类工具一样：发送按钮必须语义清楚，点击发送后输入框立即清空，Enter 直接发送，Shift+Enter 换行，底部操作和提示不能占用过多空间。
- 策划 Agent 不应该被固定模式锁死；Agent 最有价值的是智能判断任务、检索、工具调用和写入边界。前端的模式入口最多只能是任务倾向或安全约束提示，默认应交给 Agent 自主判断。
- 片状粒子的随风摆动、翻转和倾斜必须共存；不能用简单压扁、斜切或偷偷影响方向来假装翻转。
- 片状粒子的实现需要先完成清晰设计，再进行代码落地，避免继续在错误的二维变换方案上补丁式修改。
- 片状粒子的翻转和倾斜不能被做成平面内自转或只把贴图压扁；它需要真正区分运动方向、表面法线、局部边缘扭曲和受风明暗变化。
- 片状粒子不能为了表现翻转而变成奇怪的折片、灰黑碎片或不可辨认的东西；视觉上必须先像自然的花瓣、叶片或蒲公英，再谈三维姿态。
- 粒子效果不能只检查代码无报错就交付；必须真实打开页面看截图，确认视觉上不是奇怪碎片、脏点或不可辨认物，再算完成。
- 需要把后续实现设计成任务步骤，并按步骤一一完成；只有全部步骤完成、代码和验证都跑通后才能停下来。
- 樱花粒子应该优先表现“单片樱花花瓣”，不要改成五瓣樱花、五角花或普通花朵；颜色应保持柔和粉白，不能越调越灰、越脏或偏奇怪色。
- 樱花粒子不能只是静止贴图沿路径移动；必须能在真实页面上看出随风摆动、表面翻转和倾斜，且这些变化不能靠持续自转伪装。
- 樱花粒子颜色应该是低饱和、偏白的淡樱花粉，不能为了可见性调成高饱和玫粉、荧光粉或刺眼粉。
- 樱花花瓣的运动应该考虑重量、重心、角惯量和气动力矩；重量用于影响风对花瓣的推动和翻转响应大小，不能只用随机旋转或目标角度插值伪装真实随风运动。
- 樱花花瓣不能只有单点重心，还需要重量分布；实现上应能表达根部、尖端、左侧、右侧哪边更重，并让这些差异影响随风翻起、倾斜、阻尼、弯曲和视觉明暗。
- 片状粒子的重量分布不能随机乱分；同一种物质应该按轮廓体积或面积分布决定哪边更重，并且需要有密度、厚度、实际质量和重量等真实数值来区分不同物质。轻的一边应该更容易被风推动和掀起。
- 轻的一边受风影响更大不能只停留在数据上；视觉上应能看到轻边或轻端被风吹到更前面、局部抬起、错位和更灵敏摆动等自然表现。
- 粒子的出现方向需要支持多个来源，或做成矩形四条边都能单独启用并调节出现范围，用来控制粒子从哪些边、哪些区段进入画面。
- 粒子的出现应该是连续发射，不应该一阵一阵突然出现一大片。
- 片状粒子翻转时，向下运动速度不能被当成风；出生后的运动方向、下落惯性或类似重力的速度，只能影响位置移动，不能让轻端总是被判定为更竖直向下。
- 之前“头部 / 轻部往风方向偏”的效果必须用当前重量、风载、受风面和轮廓质量矩体系实现，不能靠恢复 headingRotation 或额外平面旋转伪装。
- 轻部受风效果不能只停留在调试数据非零，真实页面上必须肉眼可见轻部区域沿风向前移、抬起和随风摆动。
- 片状粒子按形状受风时，不能只是整片花瓣一起往风向移动；例如樱花花瓣遇到向右的风，尖端或更轻的连续区域应该形成局部流形式偏移，尖端那一部分更明显向右偏，主体和根部保持相对锚定。
- 这里的“尖端”只是例子，不是要程序真的预设或寻找一个固定尖端；真正目标是风根据整张贴图形状场影响局部区域，不同风向下受影响区域应随形状、质量、拖曳、边缘和薄区重新分布。
- 同类片状粒子看起来大小不同只是视觉缩放；真实物理长度、面积、体积、质量、重量、角惯量和风载基准都应该使用同一套原始材质与原始贴图数据。
- 片状粒子的形状受风不能使用显示缩放后的面积、体积或重量重新计算；显示大小只影响画面，不影响真实受风基准。
- 粒子不能全部保持竖直向下；禁止的是持续自转或强行对齐风向，不是让每片花瓣失去自己的身体朝向。
- 粒子不能只有运动方向生效；翻转、倾斜、随风摆动和基于形状场的局部受风必须在真实页面肉眼可见，不能只停留在调试数值非零。
- 粒子的摆动不能看起来像每片自己按相位摆动；主要姿态变化必须来自标准风和局部意外风的风压分量。形状场局部位移不能把透明贴图边缘拉出毛刺、尖刺或奇怪变形。
- 粒子的倾斜、翻转和 3D 效果必须能在真实页面上看出痕迹，不能只是调试数据里有数值。
- 粒子尺寸设置应是最终视觉上限；用户调了多大，粒子就不应该超过这个值。
- 粒子的出现方位 / 出现范围设置必须真实控制出生位置，不能看起来仍从默认区域或画面中间出现。
- 运行历史与版本、运行审计与上下文等附属页面不能混乱堆叠；空间分配、文字大小和信息收纳需要学习 Codex / opencode，把低频长内容放进抽屉、折叠和详情里。
- 策划 Agent 模型需要能选择或自动匹配上下文长度，并清楚显示到多少长度触发压缩，压缩逻辑要尽量接近 Codex 式上下文预算管理。
- 策划 Agent 的模型上下文可以先乐观往大估计，真实 Provider 报上下文超限后再自动调小并重试，而不是一开始保守压缩太多。
- 策划 Agent 输入框不能莫名出现内部提示、续跑 prompt 或加工后的发言；用户之前输入的发言不能因为运行中排队而停在输入框里不发送。
- 策划 Agent 线程里的用户消息也不能显示内部提示、任务倾向、目标或引用加工文本；这些只能作为提交给模型的隐藏上下文，用户可见区域必须保持用户原文。
- 策划 Agent 正在进行的任务过程不应该默认收纳藏起来；应在执行中清晰显示步骤和工具过程，等这一系列任务完成后再自动收纳。
- 人工审批点击批准后，Agent 应该继续后续任务并在界面上有状态反馈，不能批准后没有反应。
- 策划 Agent 会话必须是一等对象，不能只是隐藏的 branch 或调试分支。
- 每个策划 Agent 会话需要独立保存和展示自己的消息、运行历史、版本、审计、checkpoint 和上下文资产。
- 会话必须支持新建、选择、重命名、删除或归档、派生。
- 附属抽屉必须按“当前会话摘要、选中详情、原始证据或 JSON 二级展开”组织，不能把全局历史和低频调试信息堆成很长页面。
- 回退、回到某条消息重新编辑、派生会话和重发都必须有清晰会话语义，默认只影响当前会话。
- 策划 Agent 前端必须以“会话”为主驱动；任务、用户指令、Agent run、工具调用和 checkpoint 都是会话里的事件，不能把会话产品化成“任务线程”。
- 策划 Agent 的“智能判断”不是判断任务偏向，也不是让用户在几个模式里选；它应表示 Agent 自主判断要做什么、是否检索、是否读取文件、是否写入档案、是否审查、是否请求确认和何时收束。
- 策划 Agent 正在运行时，过程不应一开始就被收纳藏起来；执行中的步骤、工具动作、模型流和 stdout 应清楚可见，等这一系列任务完成后再自动收纳成摘要。
- 模型上下文窗口可以按模型名或 Provider 档案自动估算，但估算不能伪装成权威，也不能把 `1m` 等后缀加成不可改的模型配置；用户必须能手动覆盖上下文窗口、prompt 预算、压缩触发、输出保留和安全余量。
- 根据 1M 上下文自动推导出的 prompt、压缩触发和输出保留只是默认策略，不一定适合真实 Provider；真实超限后可以自动调小并重试，但前端必须允许用户细改这些预算。
- 策划 Agent 会话页必须立即显示用户已发送消息，不能让用户发送后看不到自己刚发了什么。
- 策划 Agent 运行完成后，过程应自动收纳成摘要或进入抽屉，不应持续占用主对话空间；正在运行时才展开显示关键步骤。
- 人工权限确认必须主动弹出或以显眼浮层提示，不应藏在输出文本或审计列表里让用户自己找停在哪里。
- 用户批准权限后 Agent 应继续原 run 或明确生成续跑入口，不能批准后仍然失败且没有可理解原因。
- 同类型权限不应每次重复询问；应参考 Codex 一类工具，支持本次允许、当前会话允许、当前小说工作区允许等授权作用域，并能在权限设置中管理。
- 副页面和抽屉不能出现文字遮挡；抽屉的关闭、收起和主要操作必须固定在可见位置，不应只有滚回顶部才可操作。
- 回退主流程应是对话级回退；更细的 checkpoint 只能作为当前最新一轮对话的详情，不应把大量回退点平铺给用户管理。
- Agent 审计和工具过程应按一次调用聚合成一条状态记录，调用成功、失败、结果和证据属于同一条记录的状态与详情，不应拆成多条占空间。
- Agent 界面命名必须清楚说明用途和影响范围，避免“高级审计”“聚焦”等不明所以的词裸露给普通工作流。
- 策划 Agent 的用户输入本质是“对话消息”，不是任务指令；回到某条消息编辑并重新发送时，当前会话视图和后续上下文必须从该消息位置重新继续，旧后续不能继续污染当前会话。
- 策划 Agent 运行完成后的过程收纳要更接近 Codex：终态只保留一行结果摘要，计划、工具、验收、checkpoint 和原始记录统一收进“过程详情”，不能完成后还把多个细节模块分散占屏。
- 策划 Agent 运行过程中也要更接近 Codex：主消息流只显示“处理中 / 耗时 / 活动行”，例如已运行命令、已读取资料、已编辑内容、已调用模型、等待确认；工具参数、stdout、计划和检查细节默认收进“展开实时细节”。
- Agent 用户界面命名不能暴露后端内部对象名；例如让 Agent 读文件时不能显示成“检索上下文资产”，应按用户动作和对象显示为“读取文件”“查找历史证据”“读取证据原文”等。
- 抽屉打开时内部折叠项不能默认全部展开；低频细节应该默认收起，并且抽屉底部需要有容易找到的关闭和收起展开项入口。
- Agent 工具过程不能把同一份资料用“工具结果、引用原文、证据读取、结果 id”多套视角重复展示；主对话里只显示读了什么、为什么读、结果是否可用，底层 JSON、run id、tool asset id 和调度层名只放调试详情。
- 当用户让 Agent 读取某个地址、文件夹或文件时，即使后端为了省上下文复用了旧运行里落盘的长内容引用，主界面也必须按用户目标显示为“读取文件内容 / 定位文件”，不能显示成“查找历史证据”“读取证据原文”这类内部调度名。
- 策划 Agent 主输入区需要支持拖拽文本文件加入当前消息上下文；拖入后应显示轻量文件 chip，不能跳转打开文件，不能把文件全文塞回输入框，也不能自动写入档案、记忆或世界书。
- 刷新页面后，当前正在运行的策划 Agent 会话必须能自动接回，不能让正在处理的对话消失；运行完成后也要像 Codex 一样在主消息下保留可扫读的一步一步过程，而不是只剩一个看不见细节的按钮。
- 一条会话正在运行时，用户也必须能自由切换到其它 Agent 会话；切换只改变当前界面焦点，不应该终止后台会话，也不应该让旧会话的运行状态锁住新会话输入区。
- 顶部鼠标悬浮唤出的页面栏和 Agent 工具栏必须限制在 Agent 工作区范围内，不能覆盖整个应用、侧栏或其它页面区域。
- 用户消息附带的拖入文件必须作为消息附件显示，并且“回到此处编辑”时要恢复这些文件 chip；不能只把文件内容藏进模型上下文后在界面里消失。
- 删除会话的确认不应该长时间阻塞整个界面；确认后应立即收起提示框，让删除在后台完成并同步会话列表。
- 完成后的 Agent 过程流应更接近 Codex：显示少量可扫读的处理段落或动作摘要，详细事件和工具流水放进过程详情，不能把大量绿色步骤平铺在主消息下。
- 每本书和每本书的 Agent 会话列表应做成右侧栏 / 右侧工作区抽屉，像左侧栏一样可隐藏、可收起，并替代单独的会话页面或临时弹窗。
- 右侧会话栏不仅负责会话选择，也可以承载少量当前小说常用功能按钮，但主消息流仍然必须占据核心空间。
- 策划 Agent 主消息区顶部和输入区不应该堆一排“会话栏 / 过程 / 会话记录 / 工作区 / 模型”等重复按钮；这些低频入口应该主要放在右侧会话栏或隐藏式入口里。
- 右侧会话栏收起入口不应该是明显占位的大竖条，而应是中线附近弱化的隐藏式悬浮小标，鼠标悬浮时再明显。
- 策划 Agent 内部滚动条不应显眼占用视觉空间；滚动可以保留，但滚动条应隐藏或极弱化。
- 侧栏信息架构必须区分全局入口和当前小说工作区：小说库、提供商、背景等全局功能不能与每本书自己的档案、世界书、记忆、扮演配置、章节写作等同级平铺；当前小说的资料体系要体现为单本书下的工作区层级。
- 侧栏关闭和打开不应该使用常驻的大按钮；更希望使用贴边隐藏式的小标，鼠标悬浮时显现并可切换。
- 关闭侧栏只能释放导航占用的空间，不能让主工作区、会话内容或其它页面内容一起消失。
- 右侧会话栏内部不要再放“收起该侧边栏”这类额外按钮；展开和收起统一使用边缘中线隐藏式小标。
- 右侧会话栏应按“当前小说 / 一本书工作区”组织：顶部是少量命令入口，下方是当前书分组，默认只展示最近几个会话，更多会话通过“展开显示”查看。
- 运行历史、上下文、文件、模型等附属页不能继续是白板式大块铺开；要和主页面同一材质、同一层级，使用紧凑分组、抽屉收纳和渐进展开。
- 右侧会话栏应该像左侧栏一样贯穿当前工作区高度，而不是只取 Agent 卡片右边的一部分。
- 策划 Agent 顶部标题栏和其它页面、抽屉不能出现纯白大条或纯白白板感；需要服从当前背景材质、透明度、模糊和轻分层。
- 附属页面和抽屉不能用大面积纯白容器挤占视图空间；首屏应更轻、更贴合主页面材质。
- 右侧会话栏贯穿不等于把中间主对话区也拉通全屏；中间应保留原本的大顶部栏和主内容边界，只由右侧会话栏单独贯穿。
- 右侧会话栏收起后必须保留贴边隐藏式小把手，不能因为侧栏自身收起而一起消失。
- 切换 Agent 会话应像 Codex 一样先在前端即时切换高亮和可见消息，再由后端补齐完整数据，不能让用户等到请求完成才看到反馈。
- 每条 Agent 会话在右侧栏里应更紧凑，默认只保留标题、最近消息和少量状态，低频操作 hover 后再出现。
- 工作区文件、运行历史、上下文、模型等附属页必须改掉纯白板和大段说明铺满首屏的问题，使用同材质、紧凑摘要、折叠和详情预览。
- 附属页面打开时不能先用深色遮罩把主界面压暗；抽屉应该是明亮、轻遮罩、可读的工作区浮层。
- 当前会话记录这类附属页不能用大 KPI 卡、三列后台报表和大空白堆叠；应该围绕当前会话记录和选中详情做更高效的两栏布局。
- 不能把“不要暗沉”修成所有附属页都是纯白板；附属页需要浅底色、弱纹理、弱边界和紧凑层级，避免大白块铺满。
- 修附属页时必须同时解决空间利用率和排版问题，不能只改颜色。
- 策划 Agent 主对话区内部不需要再显示“策划 Agent / 空闲 / 当前会话摘要”这条重复标题栏；顶部全局栏和右侧会话栏已经承担这些信息。
- 附属抽屉不应该贴满整块视窗形成大片白板，应该像浮动工作区一样有边界、留缝、紧凑宽度和渐进详情。
- 策划 Agent 的工具调用积极性需要整体加强；不能只针对拖入附件等特殊场景补丁式触发工具，而应把“是否需要调用各类工具”的判断纳入每轮 Agent 的必要程序，让 Agent 显式评估是否需要检索、读取文件、读取附件、读取历史证据、诊断、写入档案、写入记忆、编辑世界书、生成扮演配置、运行章节链路等工具，但评估不等于每轮都强制调用工具。
- 策划 Agent 不应在最开始一次性死判断是否调用工具；它应像 Codex 一样在同一轮会话中不断读取自己运行的结果、工具 observation、写入结果和失败原因，再继续决定下一步。
- 策划 Agent 如果生成了角色卡、档案、世界书、记忆、正文草稿、扮演配置等后续可复用内容，不应默认只写在回复中；应主动调用对应工具写入或给出明确不写入原因。
- 策划 Agent 的主对话展示应更像 Codex 的一段一段过程：每段是根据上一段结果继续行动，而不是把内部工具报表、协议名和调度细节堆给用户。
- 行文应该成为策划 Agent 可主动调用的工具能力，而不是只在回复里生成正文。
- 行文工具需要明确区分正常行文和扮演行文：正常行文由正文 Agent 直接依据写前定位、档案、记忆、世界书和已采纳正文慢写；扮演行文先由导演与角色 AI 形成扮演记录，再由正文 Agent 改写。
- 扮演行文和正常行文不应该混为一谈：扮演行文应先由导演与角色 AI 形成扮演记录，再由正文 Agent 改写；正常行文则应由正文 Agent 直接依据写前定位、档案、记忆、世界书和已采纳正文写草稿，并同样经过正文审查与版本记录。
- 策划 Agent 主流程不需要独立“继续”功能；如果用户想继续，只要在当前会话里直接发一句“继续”。
- 策划 Agent 发送消息后不能一直在输入框区域转圈；消息应立即清空并进入会话流，避免用户误以为没有发送。
- 策划 Agent 的附属页面和抽屉必须能正常上下滑动，不能因为固定高度或 overflow 冲突导致页面卡住。
- 策划 Agent 的运行过程要更接近 Codex 的一段一段效果：显示“已处理多久”、自然语言过程段落和轻量动作行，而不是后台工具报表。
- 策划 Agent 的单个工具失败、守卫阻断、审查阻断不应直接中断整轮流程；这些结果应像 Codex 式 observation 一样回到 Agent，让 Agent 在同一轮里继续修正、重试、请求确认或明确收束。
- 策划 Agent 的停止按钮必须即时响应；点击停止后前端应立即进入中断态，后端尽快 abort 模型调用、工具调用和子任务，不能长时间卡在等待返回。
- 页面加载不能被对话、运行历史、上下文资产或某个局部重数据拖住；应该先加载可交互的工作台外壳和当前会话摘要，再后台补齐完整状态、历史细节和重内容引用，优化视觉等待体验。
- 策划 Agent 的运行过程、工具调用、消息分段和会话交互不能只靠主观猜测设计；需要直接研究 Codex、Pi、opencode、Claude 等开源项目源码，优先借鉴它们的消息部件流、工具调用合并、事件协议和前端展示逻辑，再落到本项目。
- 策划 Agent 的目标不是口头宣称“像 Codex”，而是要用真实 API 和真实页面持续验收：底层必须能多轮模型调用、工具 observation 回流、权限批准后继续原 run、失败后给 Agent 修正机会；表层必须做到会话流、一步一步过程、工具调用与结果合并、终态自动收纳、编辑消息不新开无关会话。
- 在没有达到用户认可的 Codex 级会话体验前，不要把任务宣称完成；如果想结束，必须明确问用户当前是否已经达到预期，否则继续检查和实现可验证差距。
- 修 Codex 式 Agent 体验时不能一直围绕同几个表象问题循环；必须定位根因，例如多套过程流、旧审计 fallback、内部协议词泄漏、前端状态覆盖等，再做统一修正和验证。
